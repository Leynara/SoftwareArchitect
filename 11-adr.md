# №11: Список ADR (Architecture Decision Records)

---

**Назначение раздела:** Architecture Decision Records (ADR) — это документированные записи о ключевых архитектурных решениях, принятых в процессе проектирования системы. Каждый ADR фиксирует контекст, проблему, рассмотренные варианты, выбранное решение и его обоснование. ADR служат историей архитектурной эволюции, позволяют новым членам команды быстро вникнуть в контекст и обосновывают выбор решений перед архитектурным комитетом.

---

## 1. Структура ADR

Каждый ADR имеет следующую структуру:

| Секция | Описание |
|:---|:---|
| **Заголовок** | Краткое название решения (ADR-XXX: Название) |
| **Статус** | Предложено / Принято / Устарело / Заменено |
| **Дата** | Дата принятия решения |
| **Контекст** | Описание проблемы, требований и ограничений |
| **Рассмотренные варианты** | Альтернативные решения с их плюсами и минусами |
| **Решение** | Выбранный вариант |
| **Обоснование** | Почему выбран этот вариант (связь с бизнес-целями, НФТ, ограничениями) |
| **Последствия** | Положительные и отрицательные последствия решения |
| **Компромиссы** | Чем пришлось пожертвовать |
| **Связанные ADR** | Другие ADR, связанные с данным |
| **Связанные требования** | БТ, ФТ, НФТ |

## 2. Список ADR

### ADR-001: Выбор архитектурного стиля

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-01-15 |
| **Контекст** | Система должна масштабироваться до 50M пользователей, поддерживать независимую разработку команд (7+ команд), обеспечивать высокую доступность (99.9%) и отказоустойчивость. Компания использует multi-cloud стратегию. Требуется поддержка глобального распределения и асинхронной обработки пиковых нагрузок (марафоны, соревнования). |
| **Рассмотренные варианты** | 1. Монолит — единое приложение, простота разработки, но ограниченная масштабируемость.<br>2. Модульный монолит — лучшее разделение, но единое развертывание.<br>3. Микросервисы — максимальная гибкость, но сложность управления.<br>4. Serverless — автоматическое масштабирование, но ограничения по времени выполнения. |
| **Решение** | Микросервисная архитектура + Event-Driven (CQRS/Event Sourcing) |
| **Обоснование** | 1. Независимость команд: каждая команда отвечает за свой сервис.<br>2. Горизонтальное масштабирование: каждый сервис масштабируется отдельно.<br>3. Отказоустойчивость: сбой в одном сервисе не влияет на другие.<br>4. Multi-cloud: сервисы можно размещать в разных облаках.<br>5. Event-Driven позволяет асинхронно обрабатывать пиковые нагрузки.<br>6. CQRS/Event Sourcing обеспечивает аудит и восстановление состояния.<br>7. Соответствует бизнес-целям (БТ-1..БТ-4) и НФТ (НФТ-06, НФТ-10, НФТ-18). |
| **Последствия** | Положительные: гибкость, масштабируемость, независимость команд. Отрицательные: сложность управления распределённой системой, увеличение затрат на инфраструктуру. |
| **Компромиссы** | Сложность vs гибкость; eventual consistency vs строгая консистентность |
| **Связанные ADR** | ADR-002, ADR-003, ADR-004, ADR-005, ADR-008, ADR-010 |
| **Связанные требования** | БТ-1, БТ-2, БТ-3, БТ-4; ФТ-1, ФТ-4, ФТ-5, ФТ-9; НФТ-06, НФТ-10, НФТ-18, НФТ-19 |

### ADR-002: Выбор стиля коммуникации между сервисами

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-01-20 |
| **Контекст** | Сервисы должны обмениваться данными, сохраняя слабую связанность. Некоторые операции требуют немедленного ответа (проверка платежа, регистрация), другие могут быть асинхронными (обновление ленты, аналитика, уведомления). |
| **Рассмотренные варианты** | 1. Синхронный REST — простота, но жёсткая связанность.<br>2. Синхронный gRPC — высокая производительность, но бинарный протокол сложнее.<br>3. Асинхронный (RabbitMQ / Kafka) — слабая связанность, но eventual consistency.<br>4. Гибридный — синхронный для запросов, асинхронный для событий. |
| **Решение** | Гибридный: REST/gRPC для синхронных запросов, Apache Kafka для событий |
| **Обоснование** | 1. Kafka обеспечивает гарантированную доставку, репликацию и replay событий.<br>2. Асинхронность разгружает критические пути (запись тренировок).<br>3. REST/gRPC для операций, где нужен немедленный ответ.<br>4. Слабая связанность сервисов.<br>5. Поддержка eventual consistency (соответствует НФТ-10, НФТ-50). |
| **Последствия** | Положительные: слабая связанность, масштабируемость, надёжность. Отрицательные: увеличение задержки для асинхронных операций, сложность отладки. |
| **Компромиссы** | Простота vs асинхронность; синхронная консистентность vs производительность |
| **Связанные ADR** | ADR-001, ADR-006, ADR-010 |
| **Связанные требования** | БТ-2; ФТ-1, ФТ-4; НФТ-01, НФТ-02, НФТ-05, НФТ-10, НФТ-50 |

### ADR-003: Выбор подхода к клиент-серверному взаимодействию

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-01-22 |
| **Контекст** | Мобильные и веб-клиенты имеют разные требования к данным. Мобильные устройства требуют минимального количества запросов и агрегированных данных. Веб-админка требует детальной информации. Необходимо обеспечить безопасность (единая аутентификация). |
| **Рассмотренные варианты** | 1. Прямой вызов микросервисов — простота, но много запросов с клиента.<br>2. API Gateway — единая точка входа, маршрутизация.<br>3. API Gateway + BFF — разные BFF для разных клиентов.<br>4. GraphQL Gateway — гибкие запросы, но сложность кеширования. |
| **Решение** | API Gateway (Kong/Envoy) + BFF (Backend for Frontend) для мобильных приложений |
| **Обоснование** | 1. API Gateway обеспечивает единую точку входа, аутентификацию, rate limiting, кеширование.<br>2. BFF агрегирует данные из нескольких сервисов, уменьшая количество запросов с мобильного устройства.<br>3. Разные BFF для iOS/Android позволяют оптимизировать под платформу.<br>4. Упрощает клиентскую разработку.<br>5. Соответствует НФТ-13, НФТ-15. |
| **Последствия** | Положительные: безопасность, агрегация данных, упрощение клиентов. Отрицательные: дополнительный слой увеличивает задержку. |
| **Компромиссы** | Простота клиентов vs дополнительная задержка |
| **Связанные ADR** | ADR-001, ADR-011 |
| **Связанные требования** | БТ-3, БТ-4; ФТ-9, ФТ-10, ФТ-12; НФТ-13, НФТ-15 |

### ADR-004: Выбор стратегии хранения данных (Polyglot Persistence)

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-01-25 |
| **Контекст** | Система хранит разнородные данные: профили (транзакционные), тренировки (time-series), социальные связи (графовые), медиа (объектные), логи для поиска (текстовые). Одна БД не может эффективно обрабатывать все типы данных. |
| **Рассмотренные варианты** | 1. Одна БД (PostgreSQL) — простота, но ограниченная производительность для time-series и графов.<br>2. Две БД (PostgreSQL + Time-series) — лучше, но без поиска и графов.<br>3. Polyglot Persistence — несколько специализированных БД. |
| **Решение** | Polyglot Persistence: PostgreSQL (OLTP) + TimescaleDB (Time-series) + Elasticsearch (Поиск) + Redis (Кеш) + Neo4j (Графовые) + MinIO/S3 (Объектное) + DWH/BigQuery (Аналитика) |
| **Обоснование** | 1. Каждая БД оптимизирована под свой тип данных.<br>2. Производительность: TimescaleDB для time-series, Neo4j для графовых запросов.<br>3. Масштабируемость: каждая БД масштабируется независимо.<br>4. Соответствует НФТ-01, НФТ-05, НФТ-18, НФТ-40. |
| **Последствия** | Положительные: производительность, масштабируемость, оптимизация под типы данных. Отрицательные: сложность управления несколькими БД, необходимость синхронизации. |
| **Компромиссы** | Сложность управления vs производительность; стоимость vs эффективность |
| **Связанные ADR** | ADR-001, ADR-005 |
| **Связанные требования** | БТ-1, БТ-2; ФТ-1, ФТ-5; НФТ-01, НФТ-05, НФТ-18, НФТ-40 |

### ADR-005: Выбор стратегии шардирования данных

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-01-28 |
| **Контекст** | При росте до 50M пользователей одна БД не сможет хранить все данные. Необходимо распределение данных по узлам. Также требуется соответствие GDPR/CCPA (данные не должны покидать регион пользователя). |
| **Рассмотренные варианты** | 1. Вертикальное шардирование — разные таблицы на разных серверах.<br>2. Горизонтальное (по пользователю) — строки одной таблицы на разных серверах.<br>3. Географическое шардирование — данные хранятся в регионе пользователя.<br>4. Комбинированное — пользовательское + географическое. |
| **Решение** | Комбинированное: пользовательское шардирование + географическое шардирование |
| **Обоснование** | 1. Все данные пользователя на одном шарде (простота запросов).<br>2. Географическое шардирование обеспечивает низкую задержку и соответствие GDPR/CCPA.<br>3. Лёгкость масштабирования — добавление новых шардов.<br>4. Соответствует НФТ-06, НФТ-17, НФТ-40. |
| **Последствия** | Положительные: масштабируемость, низкая задержка, юридическое соответствие. Отрицательные: сложность кросс-шардовых запросов. |
| **Компромиссы** | Простота запросов vs сложность кросс-шардовой аналитики |
| **Связанные ADR** | ADR-004 |
| **Связанные требования** | БТ-2; ФТ-1; НФТ-06, НФТ-17, НФТ-40 |

### ADR-006: Выбор инструмента для обработки time-series данных

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-02-01 |
| **Контекст** | Система получает до 5M тренировок в день. Каждая тренировка содержит тысячи точек данных (GPS, пульс). Необходимо эффективно хранить, агрегировать и анализировать эти данные. |
| **Рассмотренные варианты** | 1. PostgreSQL с партиционированием — знакомая технология, но ограниченная производительность для time-series.<br>2. TimescaleDB — расширение PostgreSQL, поддержка SQL, сжатие до 90%.<br>3. InfluxDB — специализированная time-series БД, но сложность SQL-запросов.<br>4. Cassandra — распределённая NoSQL, но слабая поддержка time-series аналитики. |
| **Решение** | TimescaleDB + Apache Kafka для потоковой обработки |
| **Обоснование** | 1. TimescaleDB — гибрид реляционной и time-series; поддержка SQL; сжатие до 90%.<br>2. Kafka — буферизация и потоковая обработка данных с устройств.<br>3. Единый стек для команд (PostgreSQL + TimescaleDB).<br>4. Горизонтальное масштабирование через шардирование.<br>5. Соответствует НФТ-01, НФТ-02, НФТ-05. |
| **Последствия** | Положительные: производительность, масштабируемость, SQL-совместимость. Отрицательные: TimescaleDB требует тюнинга под high-load. |
| **Компромиссы** | Простота использования vs сложность настройки |
| **Связанные ADR** | ADR-001, ADR-002, ADR-004 |
| **Связанные требования** | БТ-2; ФТ-1; НФТ-01, НФТ-02, НФТ-05 |

### ADR-007: Выбор подхода к AI / ML

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-02-05 |
| **Контекст** | Система должна генерировать персонализированные тренировочные планы. На старте данных мало для качественного обучения ML. Необходимо быстро запустить MVP, а затем улучшать качество рекомендаций. |
| **Рассмотренные варианты** | 1. Rule-based система — на основе экспертных знаний, быстро, но ограниченная персонализация.<br>2. ML-модели с нуля — высокое качество, но нужны данные.<br>3. Transfer learning — адаптация существующих моделей, быстрее, чем с нуля.<br>4. Двухэтапный подход — сначала правила, потом ML. |
| **Решение** | Двухэтапный: Rule-based на MVP → ML на этапе 2 (с использованием transfer learning) |
| **Обоснование** | 1. Rule-based позволяет запустить MVP без данных для обучения.<br>2. Сбор данных с первых пользователей для последующего обучения.<br>3. Transfer learning ускоряет разработку ML-моделей.<br>4. Отдельный ML-сервис позволяет независимо масштабировать и обновлять модели.<br>5. Соответствует БТ-2 и НФТ-02. |
| **Последствия** | Положительные: быстрый MVP, сбор данных, качество на этапе 2. Отрицательные: rule-based хуже персонализирует. |
| **Компромиссы** | Качество vs скорость вывода на рынок |
| **Связанные ADR** | ADR-001 |
| **Связанные требования** | БТ-2; ФТ-4; НФТ-02 |

### ADR-008: Выбор multi-cloud стратегии

| Секция | Содержание |
|:---|:---|
| **Статус** | Принято |
| **Дата** | 2026-02-10 |
| **Контекст** | Компания не привязана к одному облачному провайдеру. 90% систем уже в облаках (AWS, GCP, Azure). Приложение должно работать в любом из них. Требуется избежать vendor lock-in. |
| **Рассмотренные варианты** | 1. Single-cloud (AWS) — простота, но vendor lock-in.<br>2. Single-cloud с резервом — основной AWS, резерв GCP, но пассивный резерв.<br>3. Multi-cloud активный — активное использование 2+ облаков, высокая сложность.<br>4. Cloud-agnostic — абстракция через Kubernetes и открытые стандарты. |
| **Решение** | Cloud-agnostic (Kubernetes + открытые стандарты) |
| **Обоснование** | 1. Возможность выбора лучшего провайдера под задачу.<br>2. Отсутствие vendor lock-in.<br>3. Kubernetes как слой абстракции.<br>4. Terraform + Crossplane для управления ресурсами во всех облаках.<br>5. Открытые стандарты: PostgreSQL, Kafka, Redis, Elasticsearch.<br>6. Соответствует НФТ-27, НФТ-28. |
| **Последствия** | Положительные: гибкость, отсутствие vendor lock-in. Отрицательные: сложность управления несколькими облаками. |
| **Компромиссы** | Гибкость vs сложность управления |
| **Связанные ADR** | ADR-001, ADR-009, ADR-010 |
| **Связанные требования** | НФТ-27, НФТ-28 |

| **Рассмотренные варианты** | 1. Самописная аутентификация (JWT + Spring Security) — полный контроль, но риск ошибок.<br>2. Keycloak — open-source, гибкий, но требует настройки.<br>3. Auth0 — SaaS, быстро, но платно при масштабировании.<br>4. AWS Cognito / GCP Identity — vendor lock-in. |
| **Решение** | OAuth 2.0 + OpenID Connect (Keycloak) + RBAC |
| **Обоснование** | 1. OAuth 2.0 и OpenID — отраслевые стандарты.<br>2. Keycloak (open-source) — бесплатно, гибкий, поддерживает social login и MFA.<br>3. RBAC обеспечивает гибкое управление правами.<br>4. Соответствие GDPR/CCPA.<br>5. Соответствует НФТ-11, НФТ-12, НФТ-13, НФТ-14, НФТ-17. |
| **Последствия** | Положительные: стандарты, гибкость, безопасность. Отрицательные: Keycloak требует настройки и поддержки. |
| **Компромиссы** | Гибкость vs усилия на поддержку |
