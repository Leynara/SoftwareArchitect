# 11. Список ADR

## ADR-001. Выбор микросервисной архитектуры

### Статус
Accepted

### Контекст
Система должна поддерживать глобальных пользователей, социальные функции, тренировки, рекомендации, интеграции с устройствами, промоакции и связь с существующими приложениями компании. Разные части системы будут развиваться разными командами и иметь разные требования к нагрузке.

### Рассмотренные варианты
1. Монолит  
2. Модульный монолит  
3. Микросервисная архитектура  

### Решение
Выбрана **микросервисная архитектура** с доменным разделением сервисов.

### Аргументы
Микросервисы позволяют:
- независимо развивать домены;
- масштабировать нагруженные части отдельно;
- изолировать сбои;
- распределять работу между командами;
- гибко подключать новые интеграции.

### Последствия
Положительные:
- высокая масштабируемость;
- автономность команд;
- независимые релизы.

Отрицательные:
- выше сложность эксплуатации;
- нужны observability, CI/CD, контрактное тестирование;
- появляется риск распределённой консистентности.

## ADR-002. Использование event-driven архитектуры

### Статус
Accepted

### Контекст
Многие действия в системе порождают цепочки последствий: завершение тренировки обновляет аналитику, достижения, уведомления, рекомендации и промо.

### Рассмотренные варианты
1. Только синхронные REST-вызовы  
2. Очереди для отдельных задач  
3. Полноценная event-driven интеграция  

### Решение
Использовать **event-driven архитектуру** для бизнес-событий.

### Ключевые события
- `training.completed`
- `achievement.unlocked`
- `challenge.completed`
- `user.joined.group`
- `inventory.replacement.suggested`
- `promotion.selected`

### Аргументы
Event-driven подход:
- снижает связанность сервисов;
- позволяет обрабатывать события асинхронно;
- повышает устойчивость к пиковым нагрузкам;
- упрощает добавление новых обработчиков.

### Последствия
Положительные:
- масштабируемость;
- гибкость;
- независимость сервисов.

Отрицательные:
- eventual consistency;
- сложность отладки;
- нужны DLQ, retry, idempotency и distributed tracing.

## ADR-003. Использование API Gateway и BFF

### Статус
Accepted

### Контекст
Клиенты системы — мобильные приложения, web, потенциально партнёрские клиенты и внутренние приложения. Им нужен единый безопасный вход в платформу.

### Решение
Использовать:
- API Gateway как единую точку входа;
- BFF для мобильных клиентов при необходимости.

### Аргументы
API Gateway обеспечивает:
- маршрутизацию;
- rate limiting;
- проверку токенов;
- централизованную защиту;
- управление версиями API.

BFF позволяет:
- адаптировать ответы под мобильный UX;
- уменьшить число запросов с клиента;
- скрыть внутреннюю сложность микросервисов.

### Последствия
Появляется дополнительный компонент, который должен быть высокодоступным и масштабируемым.

## ADR-004. Polyglot persistence

### Статус
Accepted

### Контекст
Данные системы сильно отличаются по природе:
- пользовательские профили;
- тренировки;
- телеметрия;
- социальный граф;
- события;
- аналитические агрегаты;
- промо-контент.

### Решение
Использовать разные типы хранилищ под разные задачи.

### Предлагаемые хранилища
| Тип данных | Хранилище |
|---|---|
| Пользователи, профили, настройки | Relational DB |
| Тренировки и активность | Document / wide-column DB |
| Телеметрия | Time-series storage |
| Социальные связи | Graph или специализированная модель |
| Аналитика | Data warehouse |
| Кэш | Redis / managed cache |
| События | Kafka / Pub/Sub |

### Аргументы
Единая БД для всех данных приведёт к плохой производительности, сложным схемам и дорогому масштабированию.

### Последствия
Плюсы:
- оптимизация под разные нагрузки;
- лучшая производительность;
- гибкость модели данных.

Минусы:
- сложнее эксплуатация;
- нужны data governance и стандарты доступа.

## ADR-005. Privacy by design

### Статус
Accepted

### Контекст
Система обрабатывает данные о здоровье, тренировках, маршрутах, местоположении и социальных связях. Это чувствительная информация.

### Решение
Принять **privacy by design** как обязательный архитектурный принцип.

### Основные меры
- минимизация собираемых данных;
- явное согласие пользователя;
- настройки видимости;
- маскирование геоданных;
- шифрование;
- аудит доступа;
- удаление и экспорт данных по запросу.

### Последствия
Положительные:
- снижение юридических рисков;
- рост доверия пользователей;
- готовность к глобальным рынкам.

Отрицательные:
- выше сложность реализации;
- часть маркетинговых сценариев ограничивается правилами приватности.

## ADR-006. Отдельный контур для соревнований

### Статус
Accepted

### Контекст
Соревнования могут создавать резкие пики нагрузки: регистрация участников, отправка результатов, обновление рейтингов, просмотр leaderboard.

### Решение
Выделить Competition Service и отдельный high-load контур.

### Аргументы
Соревнования имеют другую нагрузочную модель:
- burst traffic;
- read-heavy leaderboard;
- near real-time обновления;
- высокий репутационный риск при сбое.

### Последствия
Плюсы:
- изоляция нагрузки;
- возможность отдельного масштабирования;
- меньше риск деградации основных тренировочных сценариев.

Минусы:
- сложнее архитектура;
- требуется отдельное тестирование производительности.

## ADR-007. Поэтапное внедрение ML

### Статус
Accepted

### Контекст
Рекомендации важны, но на старте недостаточно данных для качественных ML-моделей.

### Решение
Начать с rule-based рекомендаций, затем перейти к ML/AI после накопления данных.

### Аргументы
На раннем этапе rule-based подход:
- быстрее;
- понятнее;
- дешевле;
- проще объясняется пользователю.

ML вводится позже для:
- персональных тренировок;
- промо;
- прогнозирования износа инвентаря;
- сегментации.

### Последствия
Снижается риск дорогостоящей, но бесполезной ML-инфраструктуры на старте.

## ADR-008. Cloud-native и Kubernetes

### Статус
Accepted

### Контекст
Компания уже активно использует облака, причём нет единственного облачного провайдера.

### Решение
Проектировать систему как cloud-native:
- контейнеризация;
- Kubernetes;
- managed services;
- infrastructure as code;
- CI/CD.

### Аргументы
Kubernetes помогает унифицировать подход к развёртыванию в multi-cloud окружении.

### Последствия
Плюсы:
- переносимость;
- стандартизированное развёртывание;
- автоматическое масштабирование.

Минусы:
- нужна зрелая platform engineering команда;
- Kubernetes сам по себе добавляет сложность.
