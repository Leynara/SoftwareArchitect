# №5: Описание рисков реализации (Risks)

---

**Назначение раздела:** Идентификация, анализ и описание рисков, связанных с реализацией проекта. Риски разделены на бизнес-риски (влияющие на достижение бизнес-целей) и технические риски (влияющие на процесс разработки, архитектуру и эксплуатацию). Для каждого риска определена вероятность наступления, степень влияния, стратегия mitigation и план действий при наступлении.

---

## 1. Методология анализа рисков

| Параметр | Описание |
|:---|:---|
| **Вероятность (P)** | Оценка вероятности наступления риска: Низкая (1–3), Средняя (4–6), Высокая (7–10) |
| **Влияние (I)** | Оценка степени влияния на проект: Низкое (1–3), Среднее (4–6), Критическое (7–10) |
| **Приоритет (Priority)** | Произведение вероятности и влияния (P × I): Низкий (1–15), Средний (16–35), Высокий (36–100) |
| **Стратегия mitigation** | Действия для снижения вероятности или влияния риска |
| **План действий при наступлении** | Что делать, если риск материализовался |

## 2. Бизнес-риски

### РБ-1: Низкое принятие геймификации и социальных функций пользователями

| Параметр | Значение |
|:---|:---|
| **Описание** | Целевая аудитория (особенно взрослые пользователи и профессионалы) может посчитать геймификацию навязчивой, "детской" или отвлекающей от реальных тренировок. Социальные функции могут не "залететь" — пользователи не будут создавать группы, участвовать в челленджах и общаться в чатах. |
| **Вероятность** | Средняя (5) |
| **Влияние** | Критическое (8) |
| **Приоритет** | Высокий (40) |
| **Связанные бизнес-цели** | БТ-1 (Удержание через социум) |
| **Связанные ФТ** | ФТ-5, ФТ-6 |

**Стратегия mitigation:**
- Проведение UX-исследований и A/B-тестирования на ранних этапах.
- Гибкие настройки: возможность отключения геймификации для тех, кому она неинтересна.
- Поэтапное внедрение: сначала базовые функции (лента, группы), потом продвинутые (челленджи, рейтинги).
- Использование поведенческой аналитики для выявления проблем на ранних стадиях.

**План действий при наступлении:**
- Сбор обратной связи через in-app опросы и интервью.
- Быстрая итерация: упрощение интерфейса, изменение механик.
- Пересмотр приоритетов: смещение фокуса на другие функции (AI-коучинг), если социальные функции не взлетают.
- Внедрение "тихих" социальных функций (например, автоматические группы на основе интересов без активного участия пользователя).

### РБ-2: Медленный рост пользовательской базы

| Параметр | Значение |
|:---|:---|
| **Описание** | Приложение не достигает целевых показателей по количеству пользователей (10M через 1 год) из-за высокой конкуренции (Strava, Nike Run Club, Garmin), слабого маркетинга или низкой виральности. Это снижает ценность социальных функций (сетевой эффект) и объём данных для ML-моделей. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Критическое (9) |
| **Приоритет** | Высокий (54) |
| **Связанные бизнес-цели** | Все (БТ-1, БТ-2, БТ-3, БТ-4) |
| **Связанные ФТ** | Все |

**Стратегия mitigation:**
- Агрессивный маркетинг при запуске: инфлюенсеры, партнёрства с фитнес-клубами.
- Программа рефералов (приведи друга — получи бонус).
- Бесплатная базовая версия с чёткими преимуществами перед конкурентами (AI-коучинг, интеграция с инвентарём).
- Интеграция с существующей аудиторией компании (покупатели товаров, пользователи других приложений).
- Поддержка импорта данных из Strava, Garmin и других приложений для снижения барьера входа.

**План действий при наступлении:**
- Анализ причин: опросы, анализ воронок, конкурентный анализ.
- Корректировка маркетинговой стратегии и ценовой модели.
- Усиление дифференцирующих функций (AI-коучинг, интеграция с инвентарём).
- Рассмотрение партнёрств с крупными платформами (Apple Health, Google Fit).

### РБ-3: Низкая конверсия в покупки (монетизация)

| Параметр | Значение |
|:---|:---|
| **Описание** | Пользователи используют приложение для трекинга, но не переходят в интернет-магазин для покупок. Целевая конверсия 18% из рекомендации в покупку не достигается. Причины: высокая цена, неудобный процесс покупки, слабые рекомендации, отсутствие доверия к бренду. |
| **Вероятность** | Средняя (5) |
| **Влияние** | Критическое (9) |
| **Приоритет** | Высокий (45) |
| **Связанные бизнес-цели** | БТ-3 (Монетизация) |
| **Связанные ФТ** | ФТ-7, ФТ-8 |

**Стратегия mitigation:**
- Тщательная проработка пользовательского пути от рекомендации до покупки (минимизация шагов, единый checkout).
- Бесшовная интеграция с существующим e-commerce (единая корзина, единый логин).
- Персонализация рекомендаций на основе данных тренировок.
- Использование виртуальных наград (монеты, скидки) для стимулирования покупок.
- A/B-тестирование ценовых моделей и промоакций.

**План действий при наступлении:**
- Анализ воронки конверсии: выявление узких мест.
- Улучшение качества рекомендаций (добавление большего числа факторов).
- Внедрение более агрессивных промоакций (скидки первым покупателям).
- Упрощение интерфейса магазина внутри приложения.

### РБ-4: Юридические риски (несоответствие GDPR, CCPA)

| Параметр | Значение |
|:---|:---|
| **Описание** | Система не соответствует требованиям защиты персональных данных в разных юрисдикциях (GDPR — Европа, CCPA — Калифорния, 152-ФЗ — Россия). Это может привести к судебным искам, штрафам (до 4% от глобального оборота по GDPR) и репутационным потерям. |
| **Вероятность** | Низкая (3) |
| **Влияние** | Критическое (10) |
| **Приоритет** | Высокий (30) |
| **Связанные бизнес-цели** | Все |
| **Связанные ФТ** | Требования безопасности (инфраструктурные) |

**Стратегия mitigation:**
- Привлечение юридической команды и внешних консультантов на этапе проектирования.
- Внедрение "Privacy by Design": шифрование данных, механизмы удаления/экспорта, согласия пользователей.
- Использование проверенных решений для управления данными (например, OneTrust).
- Регулярные аудиты соответствия.
- Географическая сегментация данных (хранение данных пользователей из ЕС только в ЕС).

**План действий при наступлении:**
- Немедленное исправление выявленных нарушений.
- Взаимодействие с регуляторами, предоставление плана корректирующих действий.
- Усиление команды по комплаенс.

## 3. Технические риски

### РТ-1: Сложность реализации офлайн-режима и синхронизации

| Параметр | Значение |
|:---|:---|
| **Описание** | Требование офлайн-записи тренировок (ФТ-1) требует сложной логики разрешения конфликтов синхронизации между локальным хранилищем на устройстве и сервером. Особенно сложны сценарии: редактирование тренировки в офлайне, одновременная запись с нескольких устройств, потеря данных при сбоях. |
| **Вероятность** | Высокая (7) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Высокий (42) |
| **Связанные бизнес-цели** | БТ-2 (AI-коучинг) |
| **Связанные ФТ** | ФТ-1 |

**Стратегия mitigation:**
- Использование проверенных паттернов: CRDT (Conflict-free Replicated Data Types) или Event Sourcing с локальным Event Store.
- Тщательное проектирование модели данных для офлайн-режима.
- Разработка прототипа на раннем этапе для проверки подходов.
- Использование библиотек и фреймворков с поддержкой офлайн-синхронизации (например, Realm, Firebase Offline).
- Ограничение функциональности в офлайн-режиме (например, запрет редактирования завершённых тренировок).

**План действий при наступлении:**
- Выявление корневых причин конфликтов.
- Внедрение дополнительной валидации и проверок.
- Возможный пересмотр модели данных.
- Увеличение времени на тестирование офлайн-сценариев.

### РТ-2: Низкое качество ML-моделей для AI-коучинга

| Параметр | Значение |
|:---|:---|
| **Описание** | ML-модели для генерации тренировочных планов и рекомендаций требуют большого объёма качественных данных. На старте данных мало, что может привести к низкому качеству рекомендаций. Пользователи разочаруются в AI-коучинге, что снизит доверие к приложению. |
| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Высокий (48) |
| **Связанные бизнес-цели** | БТ-2 (AI-коучинг) |
| **Связанные ФТ** | ФТ-4 |

**Стратегия mitigation:**
- На MVP использование rule-based системы (на основе экспертных знаний), а не ML.
- Сбор данных с первых пользователей для обучения моделей (с их согласия).
- Использование transfer learning (адаптация существующих моделей из открытых источников).
- Привлечение спортивных физиологов и тренеров для валидации рекомендаций.
- Постоянный мониторинг качества рекомендаций и A/B-тестирование.

**План действий при наступлении:**
- Откат на rule-based версию для критических пользователей.
- Увеличение времени на сбор данных перед внедрением ML.
- Усиление команды Data Science.
- Пересмотр метрик качества моделей.

### РТ-3: Сложность интеграции с разнообразными внешними устройствами

| Параметр | Значение |
|:---|:---|
| **Описание** | Поддержка множества производителей (Garmin, Polar, Suunto, Wahoo, Apple Watch и т.д.) с разными протоколами (BLE, ANT+), форматами данных и версиями API — это сложная инженерная задача. Некоторые устройства имеют закрытые API, другие — часто меняются, что требует постоянного сопровождения. |
| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Высокий (40) |
| **Связанные бизнес-цели** | БТ-2 (AI-коучинг) |
| **Связанные ФТ** | ФТ-2 |

**Стратегия mitigation:**
- Создание абстрактного слоя для унификации работы с разными устройствами (Device Adapter Pattern).
- Использование стандартных протоколов и библиотек (Android BLE, CoreBluetooth для iOS).
- Приоритизация поддержки наиболее популярных устройств (Garmin, Apple Watch, Polar) на MVP.
- Партнёрство с производителями для получения доступа к их API и SDK.
- Автоматизированное тестирование интеграций.

**План действий при наступлении:**
- Быстрое реагирование на изменения API производителей (выделение команды на поддержку).
- Рассмотрение использования готовых решений (например, Fitbit API, Google Fit).
- Упрощение модели данных для унификации.

### РТ-4: Проблемы с масштабируемость при пиковых нагрузках

| Параметр | Значение |
|:---|:---|
| **Описание** | Во время популярных спортивных событий (марафоны, крупные соревнования) нагрузка на систему может резко возрасти (10,000+ RPS). Недостаточное масштабирование может привести к отказу системы, потере данных тренировок и негативному пользовательскому опыту. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Критическое (8) |
| **Приоритет** | Высокий (48) |
| **Связанные бизнес-цели** | Все |
| **Связанные ФТ** | ФТ-1 (запись тренировок) |

**Стратегия mitigation:**
- Проектирование системы с горизонтальным масштабированием.
- Использование асинхронной обработки (Kafka) для разгрузки критических путей.
- Автоматическое масштабирование (Kubernetes HPA) на основе нагрузки.
- Регулярное нагрузочное тестирование.
- Планирование ёмкости (capacity planning) на основе прогнозов.

**План действий при наступлении:**
- Автоматическое масштабирование дополнительных ресурсов.
- Включение защитных механизмов (rate limiting, circuit breaker) для предотвращения каскадного отказа.
- Переключение трафика на резервные регионы.
- Анализ инцидента и корректировка планов масштабирования.

### РТ-5: Утечка или компрометация данных пользователей

| Параметр | Значение |
|:---|:---|
| **Описание** | Хакерская атака, внутренняя ошибка или уязвимость могут привести к утечке персональных данных (PII) и данных о здоровье (PHI). Это влечёт юридические последствия (штрафы, иски), репутационные потери и отток пользователей. |
| **Вероятность** | Низкая (3) |
| **Влияние** | Критическое (10) |
| **Приоритет** | Высокий (30) |
| **Связанные бизнес-цели** | Все |
| **Связанные ФТ** | Требования безопасности (инфраструктурные) |

**Стратегия mitigation:**
- Шифрование данных в покое (AES-256) и транзите (TLS 1.3).
- Использование безопасных практик аутентификации (OAuth 2.0, MFA).
- Регулярные пентесты и аудиты безопасности.
- Мониторинг угроз и алертинг на аномальное поведение.
- Обучение команды безопасным практикам разработки (DevSecOps).
- Использование облачных сервисов с высоким уровнем безопасности (AWS, GCP).

**План действий при наступлении:**
- Немедленное отключение уязвимых систем.
- Расследование инцидента (привлечение внешних экспертов).
- Уведомление пользователей и регуляторов (согласно законодательству).
- Усиление мер безопасности.
- Компенсационные меры для пострадавших пользователей.

### РТ-6: Проблемы интеграции с legacy-системами компании

| Параметр | Значение |
|:---|:---|

| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (6) |

| **Связанные бизнес-цели** | БТ-3 (Монетизация) |
| **Связанные ФТ** | ФТ-7, ФТ-8 |

**Стратегия mitigation:**
- Создание Anti-Corruption Layer (ACL) для изоляции микросервисов от legacy.
- Использование API Gateway с трансформацией запросов/ответов.
- Версионирование API и постепенный переход на новые версии.
- Тесное взаимодействие с командами, поддерживающими legacy-системы.
- Поэтапная интеграция: сначала критичные функции (аутентификация, заказы), потом остальные.

**План действий при наступлении:**
- Выявление причин сбоев и устранение.
- Усиление команды интеграции.
- Рассмотрение альтернативных подходов (например, ручной импорт данных до стабилизации интеграции).
- Пересмотр графика интеграции.

### РТ-7: Недостаточная наблюдаемость и сложность диагностики проблем

| Параметр | Значение |
|:---|:---|
| **Описание** | В распределённой микросервисной системе сложно диагностировать проблемы без правильно настроенных инструментов мониторинга, логирования и трассировки. Это увеличивает время восстановления (MTTR) и может привести к длительным простоям. |
| **Вероятность** | Средняя (5) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Средний (30) |
| **Связанные бизнес-цели** | Все |
| **Связанные ФТ** | Все |

**Стратегия mitigation:**
- Внедрение инструментов наблюдаемости с самого начала (Prometheus, Grafana, ELK, Jaeger).
- Единая система логирования с корреляционными ID.
- Метрики SLA/SLO и алертинг.
- Создание Runbooks для типовых инцидентов.
- Регулярные тренировки команды по реагированию на инциденты (Chaos Engineering).

**План действий при наступлении:**
- Дооснащение системы инструментами наблюдаемости.
- Улучшение документации и Runbooks.
- Привлечение внешних экспертов для аудита наблюдаемости.

### РТ-8: Высокая стоимость инфраструктуры при росте данных

| Параметр | Значение |
|:---|:---|
| **Описание** | Хранение и обработка больших объёмов данных (GPS-треки, метрики, медиа) может привести к неконтролируемому росту затрат на инфраструктуру (cloud costs). Особенно критично для time-series данных (TimescaleDB) и объектного хранения (S3). |
| **Вероятность** | Высокая (7) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Высокий (42) |
| **Связанные бизнес-цели** | Все |
| **Связанные ФТ** | ФТ-1 (запись тренировок) |

**Стратегия mitigation:**
- Использование компрессии данных (Parquet, сжатие time-series).
- Автоматическое архивирование старых данных в холодное хранилище (Glacier / Deep Archive).
- Правильная политика хранения (TTL, удаление устаревших данных).
- Оптимизация запросов и использование индексов.
- Мониторинг затрат в реальном времени (Cloud Cost Management).
- Использование spot-инстансов для некритичных вычислений.

**План действий при наступлении:**
