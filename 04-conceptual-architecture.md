# №4: Разработка концептуальной архитектуры (Conceptual Architecture)

---

**Назначение раздела:** Концептуальная архитектура определяет высокоуровневую структуру системы, её основные компоненты, их взаимодействие и ключевые архитектурные стили. Она служит основой для последующей детальной проработки и позволяет согласовать видение системы между стейкхолдерами на раннем этапе. Концептуальная архитектура описывает "что" делает система и "как" она организована, без погружения в детали реализации.

---

## 1. Бизнес-контекст и стратегические цели

Система представляет собой глобальную спортивную платформу, которая объединяет:

- **Трекинг и аналитику** — сбор данных с носимых устройств и смартфонов, их обработку, хранение и визуализацию.
- **Социальную экосистему** — сообщества, чаты, друзья, совместные тренировки.
- **Геймификацию** — уровни, достижения, челленджи.
- **AI-коучинг** — персонализированные планы и рекомендации.
- **Монетизацию** — бесшовные покупки, подписки, промоакции.
- **B2B-платформу** — инструменты для региональных партнёров (фитнес-клубы, организаторы событий).

Ключевые бизнес-цели (БТ-1..БТ-4) требуют:
- Глобальной доступности и масштабируемости.
- Надёжности и отказоустойчивости.
- Соответствия требованиям безопасности и приватности (GDPR, CCPA).
- Multi-cloud стратегии (отсутствие привязки к одному провайдеру).
- Лёгкой интеграции с существующими системами компании (legacy e-commerce).

## 2. Архитектурный стиль и подход

**Выбранный стиль: Микросервисная архитектура + Event-Driven (CQRS/Event Sourcing)**

**Обоснование:**

| Требование | Почему микросервисы + Event-Driven |
|:---|:---|
| **Глобальная масштабируемость** | Возможность независимого масштабирования компонентов в зависимости от нагрузки (например, сервис трекинга — высокие пиковые нагрузки, сервис B2B — равномерная нагрузка) |
| **Multi-cloud стратегия** | Каждый микросервис может быть размещён в наиболее подходящем облаке (AWS для вычислений, GCP для ML, Azure для интеграций) |
| **Независимость команд** | Разные команды разработчиков могут работать над сервисами независимо (Social, AI, Payments, B2B) |
| **Отказоустойчивость** | Сбой в одном сервисе (например, AI) не влияет на работу других (трекинг продолжает работать) |
| **Интеграция с legacy** | Антикоррупционный слой (Anti-Corruption Layer) изолирует микросервисы от устаревших систем |
| **Глобальное распределение данных** | Событийный подход (Event Sourcing) позволяет синхронизировать данные между регионами (eventual consistency) |
| **Масштабируемость AI-вычислений** | ML-сервисы могут быть изолированы и масштабироваться отдельно, с использованием GPU-инстансов |

## 4. Основные компоненты системы

### 4.1. Клиентский слой (Client Layer)

| Компонент | Назначение | Ключевые особенности |
|:---|:---|:---|
| **Мобильное приложение (iOS/Android)** | Основной интерфейс для пользователей | Нативная разработка (Swift/Kotlin) + React Native для административных частей; поддержка офлайн-режима; локальное кеширование |
| **Веб-приложение (Partner Portal)** | Интерфейс для партнёров (фитнес-клубы, организаторы) | React/Next.js; адаптивный дизайн; дашборды с аналитикой |
| **Внешние устройства (BLE/ANT+)** | Пульсометры, GPS-часы, умные весы | Поддержка стандартных протоколов; автоматическое обнаружение |

### 4.2. Слой доступа и маршрутизации (Edge Layer)

| Компонент | Назначение | Ключевые особенности |
|:---|:---|:---|
| **API Gateway (Kong/Envoy)** | Единая точка входа для всех клиентских запросов | Маршрутизация, аутентификация (OAuth 2.0), rate limiting, кеширование, логирование |
| **BFF (Backend for Frontend)** | Адаптация API под мобильные устройства | Агрегация данных из нескольких сервисов; минимизация числа запросов с устройства |

### 4.3. Бизнес-сервисы (Business Services)

#### 4.3.1. Сервис Users & Profiles (Управление пользователями)

| Ответственность | Ключевые функции |
|:---|:---|
| Управление профилями пользователей | Регистрация, аутентификация, профили, настройки приватности |
| Управление инвентарём | Добавление обуви, снарядов, отслеживание пробега |
| Предпочтения и цели | Виды спорта, уровень подготовки, цели |

#### 4.3.2. Сервис Social (Социальные функции)

| Ответственность | Ключевые функции |
|:---|:---|
| Социальные группы | Создание, поиск, вступление, модерация |
| Друзья и подписки | Заявки, лента активности, уведомления |
| Чаты и общение | Личные и групповые чаты, обмен медиа |
| Поиск людей | По интересам, геолокации, маршрутам |

#### 4.3.3. Сервис Tracking (Трекинг тренировок) — Event-Driven

| Ответственность | Ключевые функции |
|:---|:---|
| Сбор данных тренировок | GPS, пульс, темп, мощность, высота |
| Обработка в реальном времени | Вычисление метрик, детекция рекордов |
| Офлайн-синхронизация | Приём данных с устройства при восстановлении сети |
| Генерация событий | Тренировка завершена, рекорд побит, износ инвентаря |

#### 4.3.4. Сервис AI / ML (Коучинг)

| Ответственность | Ключевые функции |
|:---|:---|
| Генерация планов | На основе целей, уровня подготовки, истории |
| Анализ данных | Прогнозирование прогресса, выявление перетренера |
| Рекомендации | Замена инвентаря, корректировка нагрузок |

#### 4.3.5. Сервис Gamification (Геймификация)

| Ответственность | Ключевые функции |
|:---|:---|
| Система уровней и XP | Начисление опыта, переход между уровнями |
| Достижения | Значки за события (марафон, стрик, челлендж) |
| Челленджи | Индивидуальные и командные соревнования |
| Виртуальные награды | Внутренняя валюта, обмен на скидки |

#### 4.3.6. Сервис B2B / Partners (Партнёрская платформа)

| Ответственность | Ключевые функции |
|:---|:---|
| Региональные промоакции | Таргетинг по геолокации, видам спорта |
| Админ-панель | Самостоятельное управление акциями, статистика |
| Публичное API | Интеграция с партнёрскими системами |
| События | Календарь забегов, регистрация |

#### 4.3.7. Сервис Commerce & Monetization (Монетизация)

| Ответственность | Ключевые функции |
|:---|:---|
| Подписки | Freemium-модель, управление подписками |
| Платежи | Интеграция с Stripe, Apple Pay, Google Pay |
| Интеграция с e-commerce | Бесшовный переход в магазин компании, единая корзина |
| Legacy-адаптация | Антикоррупционный слой для старых систем |

### 4.4. Слой данных (Data Layer)

| Хранилище | Назначение | Ключевые характеристики |
|:---|:---|:---|
| **PostgreSQL (OLTP)** | Транзакционные данные: профили, заказы, подписки | ACID, репликация, шардирование по регионам |
| **TimescaleDB** | Time-series данные: тренировки, показатели датчиков | Эффективное хранение, агрегация по времени |
| **Elasticsearch** | Поиск пользователей, групп, событий | Полнотекстовый поиск, гео-поиск |
| **Redis** | Кеширование, сессии, очереди | In-memory, низкая задержка |
| **Neo4j (графовая БД)** | Социальные связи (друзья, группы, интересы) | Эффективные запросы графов (например, "друзья друзей") |
| **MinIO / S3** | Хранение медиа-файлов (фото, аватары) | Объектное хранилище, CDN-интеграция |
| **DWH / BigQuery** | Аналитика, BI-отчёты, обучение моделей | Колоночное хранение, быстрые аналитические запросы |

### 4.5. Интеграционные слои

| Компонент | Назначение | Ключевые характеристики |
|:---|:---|:---|
| **Message Broker (Apache Kafka)** | Асинхронный обмен событиями между сервисами | Event Sourcing, гарантированная доставка, Replay |
| **Event Store** | Хранение всех событий для аудита и восстановления | Immutable log, возможность восстановления состояния |
| **Anti-Corruption Layer** | Адаптация legacy-систем компании | Трансформация данных, изоляция бизнес-логики |
| **Integration Hub (MuleSoft / Custom)** | Централизованные интеграции с внешними системами | Погода, карты, партнёрские API |

## 5. Ключевые архитектурные паттерны

| Паттерн | Применение | Обоснование |
|:---|:---|:---|
| **Microservices** | Все бизнес-сервисы | Независимое масштабирование, развитие, развертывание |
| **Event Sourcing + CQRS** | Сервис Tracking, Social, Gamification | Асинхронность, аудит, восстановление состояния, масштабируемость чтения и записи |
| **API Gateway + BFF** | Входной слой | Упрощение клиентов, безопасность, маршрутизация |
| **Circuit Breaker** | Интеграции с внешними системами (погода, карты) | Защита от каскадных отказов |
| **Saga Pattern** | Транзакции между сервисами (покупка + подписка) | Распределённые транзакции без 2PC |
| **CQRS (Command Query Responsibility Segregation)** | Сервисы с высокой нагрузкой на чтение (поиск, аналитика) | Разделение моделей чтения и записи |
| **Eventual Consistency** | Распределённые данные | Согласованность в рамках глобальной системы |
| **Retry + Exponential Backoff** | Взаимодействие с внешними API | Устойчивость к временным сбоям |
| **Bulkhead Pattern** | Изоляция ресурсов сервисов | Защита от перегрузки одного компонента |
| **Feature Toggles** | Постепенный релиз функций | Гибкость, A/B тестирование |

## 6. Интеграция с внешними системами

| Внешняя система | Назначение | Протокол | Примечание |
|:---|:---|:---|:---|
| **Legacy e-commerce (Существующий магазин)** | Покупка товаров | REST API (SOAP) | Антикоррупционный слой, трансформация данных |
| **Существующие приложения компании** | Единая аутентификация, общие данные | REST / GraphQL | Версионирование API, обратная совместимость |
| **Платёжные системы (Stripe, Braintree)** | Приём платежей, управление подписками | REST API | PCI DSS соответствие |
| **Wearable APIs (Garmin, Polar, Suunto)** | Импорт данных тренировок | REST / OAuth | Автоматическая синхронизация |
| **Apple Health / Google Fit** | Импорт шагов, сна, пульса | HealthKit / Google Fit API | Разрешения пользователя |
| **Картографические сервисы (Mapbox, Google Maps)** | Отображение маршрутов, геокодинг | REST / SDK | Кеширование карт для офлайн-режима |
| **Погодные сервисы** | Учёт погоды в рекомендациях | REST API | Кеширование на 1 час |
| **Push-сервисы (Firebase, APNS)** | Уведомления | HTTP/2 | Сегментация по предпочтениям |

## 7. Масштабирование и производительность

### 7.1. Стратегии масштабирования

| Компонент | Стратегия | Подход |
|:---|:---|:---|
| **API Gateway** | Горизонтальное масштабирование | Запуск нескольких экземпляров с балансировкой через DNS/ALB |
| **Tracking Service** | Горизонтальное масштабирование (операции записи) | Шардирование данных по пользователям (consistent hashing) |
| **AI Service** | Горизонтальное масштабирование вычислений | Использование GPU-инстансов с очередью запросов (Kafka) |
| **PostgreSQL** | Вертикальное + Горизонтальное (Read Replicas) | Master-slave; шардирование по регионам |
| **TimescaleDB** | Горизонтальное масштабирование (Partitioning) | Партиционирование по времени и пользователю |
| **Elasticsearch** | Горизонтальное масштабирование | Шардирование индексов, репликация |
| **Redis** | Кластеризация | Redis Cluster с шардированием ключей |

### 7.2. Ожидаемые нагрузки (Целевые показатели)

| Показатель | Значение | Комментарий |
|:---|:---|:---|
| **Количество пользователей** | 10M (через 1 год), 50M (через 3 года) | Рост за счёт виральности и маркетинга |
| **Пиковые RPS (запросов в секунду)** | 10,000 | Во время популярных событий (марафоны) |
| **Количество тренировок в день** | 5M (50M пользователей × 0.1 тренировки/день) | Усреднённая активность |
| **Объём данных в день** | ~500 ГБ (GPS-треки, фото, метрики) | Необходимо архивирование в холодное хранилище |
| **Время отклика (API)** | < 200 мс (p95) | Для UI-запросов; асинхронные — до 2 с |
| **Доступность (SLA)** | 99.9% | Допускается 8.7 часов простоя в год |

## 8. Атрибуты качества, обеспеченные архитектурой

| Атрибут качества | Реализация в архитектуре |
|:---|:---|
| **Масштабируемость** | Микросервисы + горизонтальное масштабирование; шардирование БД; Event-Driven для асинхронной обработки |
| **Надёжность** | Event Sourcing + Replay; Circuit Breaker; Bulkhead; Health Checks; Graceful Degradation |

| **Адаптивность** | Feature Toggles; A/B тестирование; CI/CD; Canary Releases |
| **Переносимость (Multi-cloud)** | Kubernetes + Terraform; использование открытых стандартов (PostgreSQL, Kafka, Redis) |
| **Производительность** | Кеширование (Redis); CQRS для читающих операций; асинхронная обработка через Kafka |

## 9. Безопасность на концептуальном уровне

| Аспект | Реализация |
|:---|:---|
| **Аутентификация** | OAuth 2.0 + OpenID Connect (Keycloak / Auth0) |
| **Авторизация** | RBAC (роли: пользователь, модератор, партнёр, администратор) |
| **Шифрование данных** | TLS 1.3 (транзит), AES-256 (покой), шифрование на уровне приложения для PII/PHI |
| **Защита API** | Rate Limiting, IP Whitelisting (для партнёров), JWT валидация |
| **Аудит** | Логирование всех критических действий (Event Store) |
| **Соответствие** | GDPR, CCPA, 152-ФЗ (механизмы "права на забвение", экспорт данных) |
| **DDoS-защита** | Cloud-native WAF (AWS Shield / Cloudflare), auto-scaling |

## 12. Компромиссы и допущения в концептуальной архитектуре

| Компромисс | Решение | Причина |
|:---|:---|:---|
| **Консистентность vs Производительность** | Eventual Consistency; асинхронное обновление данных | Глобальное распределение требует компромисса; высокая производительность важнее синхронной консистентности |
| **Сложность vs Гибкость** | Микросервисы (сложнее монолита) | Требование независимой разработки команд; возможность постепенной эволюции |
| **Multi-cloud vs Простота** | Kubernetes + абстрактные сервисы | Нет привязки к одному провайдеру; сложность управления оправдана бизнес-требованием |
| **Офлайн-режим vs Синхронизация** | Локальная БД + CRDT/Event Sourcing | Пользователи тренируются без интернета; сложность реализации оправдана критичностью для пользователей |
| **AI-коучинг vs Время разработки** | Rule-based MVP, ML-планы позже | Первая версия AI может быть упрощённой; глубокие модели требуют данных и времени |
