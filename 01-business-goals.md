# №1: Бизнес-цели (Business Goals)

## Общее описание
Компания стремится создать глобальное спортивное уникальное приложение, которое не только отслеживает активность, но и формирует вокруг бренда живую экосистему. Приложение должно стать ежедневным спутником для спортсменов любого уровня — от новичков до профи, — помогая им достигать целей, покупать правильные товары и находить единомышленников по всему миру.

---

## Бизнес-цель 1. Глобальное удержание аудитории через социальную экосистему и геймификацию

### Описание
Приложение должно стать не просто утилитой, а социальной платформой, где пользователи находят единомышленников, соревнуются, поддерживают друг друга и чувствуют принадлежность к большому сообществу. Это достигается через микро-сообщества (по локациям, интересам, уровню подготовки), командные челленджи и систему рейтингов. Основной фокус — на внутренней мотивации («соревнование с собой»), но с элементами здоровой конкуренции.

### Обоснование
Главная проблема фитнес-приложений — высокий отток после 3–6 месяцев (Churn Rate до 70%). Причины: потеря мотивации, одиночество, отсутствие новых целей. Социальные связи и геймификация создают «эффект привязки»: пользователи возвращаются не только ради тренировки, но и чтобы пообщаться, проверить ленту друзей, не дать команде проиграть в челлендже. Это формирует ежедневную привычку и повышает порог выхода.

### Ожидаемый эффект
- Снижение оттока (Churn) с 70% до 45% к концу первого года.
- Рост DAU/MAU (Daily Active Users / Monthly Active Users) до 35% (признак «липкого» приложения).
- Увеличение среднего времени сессии с 5 до 12 минут.

### Метрики успеха
| Метрика | Целевое значение |
| :--- | :--- |
| Monthly Churn Rate | ≤ 7% через 6 месяцев после релиза |
| Коэффициент виральности (K-factor) | ≥ 0.6 (каждый пользователь приводит минимум 0.6 нового) |
| Количество активных групп/челленджей на пользователя | ≥ 2 |
| Retention D30 (доля вернувшихся на 30-й день) | ≥ 40% |

### Связанные продуктовые возможности
- Социальные группы по интересам.
- Поиск людей рядом и по интересам.
- Уведомления о достижениях друзей.
- Уведомления о результатах професиионалов.
- Геймификация: уровни, ачивки, челленджи.

## Бизнес-цель 2. Персонализированный AI-коучинг как основа долгосрочной ценности

### Описание
Приложение выступает в роли «цифрового тренера», который анализирует данные с носимых устройств (пульс, сатурация, темп), историю активности, качество сна и даже погоду, чтобы динамически корректировать тренировочный план. План адаптируется в реальном времени: если пользователь перетренирован — предлагается восстановительная сессия; если прогресс замедлился — меняется нагрузка. Это выводит приложение из категории «записная книжка» в категорию «незаменимый помощник».

### Обоснование
Пользователи платят за результат, а не за функции. Без умной адаптации планы становятся бесполезными через 2–3 недели. Внедрение AI-коучинга — это наш главный «удерживающий крючок» для продвинутой аудитории (которая генерирует основной ворд-оф-мус). Кроме того, это создаёт высокий барьер для копирования конкурентами (нужны объёмы данных и обученные модели).

### Ожидаемый эффект
- Рост завершаемости тренировочных планов с 30% до 75%.
- Повышение лояльности среди «марафонцев» и «железных людей» (B2C-сегмент с высоким LTV).
- Сбор уникальных биомедицинских данных, которые можно монетизировать (анонимизированно) для R&D компании.

### Метрики успеха
| Метрика | Целевое значение |
| :--- | :--- |
| Процент пользователей, выполнивших план на 80%+ | ≥ 60% |
| Средний прирост VO2max (оценка) у пользователей за 3 месяца | ≥ 5% |
| Доля пользователей, подключивших 2+ внешних устройства | ≥ 30% от активной базы |
| Индекс удовлетворённости коучингом (опрос) | ≥ 4.5 из 5 |

### Связанные продуктовые возможности
- AI-формирование планов и расписания.
- Интеграция со сторонними датчиками и фитнес-функциями телефона.
- Глубокое сравнение с собой, друзьями и профи.

## Бизнес-цель 3. Монетизация через лояльность и бесшовную интеграцию с продажами

### Описание
Сделать приложение не просто трекером активности, а «магазином внутри кармана», где покупка товара — это логичное продолжение тренировочного процесса. Когда беговые кроссовки пользователя изнашиваются (по данным датчиков и пробега), приложение не просто уведомляет об этом, а предлагает купить новые с персональной скидкой, показывая, почему именно эта модель подходит под его стиль бега (пронацию, темп). Аналогично — для одежды по сезону или питания.

### Обоснование
Рынок спортивных товаров перенасыщен, и бренды конкурируют за долю кошелька. Традиционная реклама (баннеры, email-рассылки) имеет низкую конверсию (~2–3%). Персонализированная рекомендация в момент, когда у пользователя возникает *реальная потребность* (например, кроссовки «умерли» после 800 км), даёт конверсию в 15–25%. Кроме того, мы снижаем трение в покупке: пользователю не нужно уходить из приложения — интеграция с существующим магазином происходит в фоновом режиме.

### Ожидаемый эффект
- Повышение среднего чека за счёт кросс-продаж (кроссовки + носки + гель для питания).
- Сокращение цикла принятия решения о покупке с 3 дней до 30 минут.
- Увеличение LTV (Lifetime Value) пользователя на 40% за счёт регулярных умных рекомендаций.

### Метрики успеха
| Метрика | Целевое значение |
| :--- | :--- |
| Конверсия из рекомендации в покупку | ≥ 18% в первый год |
| Доля выручки мобильного приложения в общих онлайн-продажах | 35% к концу 2-го года |
| Repeat Purchase Rate (повторные покупки) | +25% по сравнению с пользователями не из приложения |
| NPS (Net Promoter Score) по разделу «Покупки» | ≥ 60 |

### Связанные продуктовые возможности
- Учёт инвентаря + автоматические оповещения об износе.
- Персонализированные промоакции и новости.
- Интеграция с существующим e-commerce бэкендом (требование №10 из вводной).

## Бизнес-цель 4. Создание платформы для локального спортивного маркетинга (B2B2C)

### Описание
Приложение становится витриной для региональных фитнес-клубов, тренеров, организаторов забегов и локальных магазинов спортивного питания. Специализированный кабинет для партнёров позволяет им создавать таргетированные промоакции (только для бегунов в Москве, только для йогов в Сан-Франциско), набирать группы на тренировки, продавать абонементы. Компания берёт комиссию с каждой транзакции или плату за размещение (SaaS).

### Обоснование
Требование «вставки региональных промоакций» — это не просто фича, а полноценный бизнес-канал. Крупные бренды спортивных товаров уже давно интегрируются с локальными экосистемами. Мы даём партнёрам доступ к нашей аудитории с высоким уровнем вовлечённости (спортсмены), а пользователям — полезный контент (скидки, события). Это диверсифицирует доходы компании, делая их независимыми от сезонности продаж одежды.

### Ожидаемый эффект
- Формирование нового стабильного источника дохода (до 15% от общего дохода компании через 3 года).
- Увеличение количества офлайн-событий (забегов, марафонов) под брендом компании, что укрепляет имидж.
- Сбор данных о локальных предпочтениях для улучшения ассортимента продукции в регионах.

### Метрики успеха
| Метрика | Целевое значение |
| :--- | :--- |
| Количество активных региональных партнёров | ≥ 500 через 1 год |
| Доход от партнёрской платформы (GMV) | $10M к концу 2-го года |
| Конверсия пользователя в событие (регистрация на забег/тренировку) | ≥ 8% |
| Количество промоакций, запущенных в месяц | ≥ 1000 |

### Связанные продуктовые возможности
- Новости спорта и промоакции с региональной привязкой.
- Возможность интеграции с внешними системами бронирования (календари, оплата).
- Админ-панель для партнёров (отдельный модуль, не входящий в клиентское приложение, но требующий API).

## Резюме бизнес-целей (для принятия инвестиционного решения)

| Приоритет | Цель | Ключевая метрика успеха | Риск недостижения |
| :---: | :--- | :--- | :--- |
| **1** | **Удержание через социум** | Churn ≤ 7% в месяц | Сообщества не "залетают", геймификация кажется навязанной. |
| **2** | **AI-коучинг** | Завершаемость планов ≥ 60% | Недостаточно данных или низкое качество ML-моделей. |
| **3** | **Монетизация через продажи** | Конверсия из рекомендации ≥ 18% | Слабый алгоритм рекомендаций, некачественный инвентарный учёт. |
| **4** | **B2B-платформа** | 500+ партнёров в год | Сложность юридических договоров в разных странах, малый спрос. |

## Ограничения и допущения

### Ограничения (Constraints)

| № | Ограничение | Описание | Влияние на архитектуру |
| :---: | :--- | :--- | :--- |
| **C-01** | **Регуляторные требования** | Обязательное соответствие GDPR (Европа), CCPA (Калифорния), а также локальным законам о хранении персональных данных (например, 152-ФЗ в России). | Требуется гибкая система управления данными: возможность географической сегментации хранения, механизмы "права на забвение" и экспорта данных. |
| **C-02** | **Мульти-облачная стратегия** | Компания не привязана к одному провайдеру (AWS, GCP, Azure используются ситуационно). Приложение должно быть cloud-agnostic. | Использование абстрактных интерфейсов для инфраструктуры (Kubernetes, Terraform), избегание проприетарных сервисов (например, DynamoDB, Cloud Spanner) в пользу открытых стандартов (PostgreSQL, Cassandra, Kafka). |
| **C-03** | **Интеграция с legacy-системами** | Существующие e-commerce приложения и узкоспециализированные спортивные приложения имеют свои API, часто устаревшие (SOAP, REST v1) и с разными моделями данных. | Необходим слой адаптации (API Gateway + BFF) для трансформации данных. Нельзя переписывать legacy — только интеграция. |
| **C-04** | **Бюджетные ограничения** | На первый год выделен бюджет $5M на разработку и инфраструктуру. | Принятие компромиссных решений: часть функций (например, продвинутый AI-коучинг) может быть реализована во второй очереди, а не в MVP. |
| **C-05** | **Сроки вывода на рынок** | MVP должен быть запущен не позднее чем через 9 месяцев с момента старта разработки. | Приоритизация функциональности. В первую очередь реализуются ядро трекинга, социальный базовый слой и интеграция с датчиками. Монетизация и B2B-платформа — во второй очереди. |
| **C-06** | **Языковой и культурный барьер** | Команды разработки распределены по разным странам (США, Европа, Азия). Код и документация — на английском. | Единый стандарт кодирования, CI/CD, обязательное покрытие кода тестами. Использование API-first подхода для параллельной разработки модулей. |
| **C-07** | **Поддержка офлайн-режима** | Пользователи могут тренироваться в местах без устойчивого интернета (горы, леса, подземные паркинги). | Приложение должно кешировать данные локально и синхронизироваться при восстановлении соединения. Требуется локальная база данных на устройстве (SQLite/Realm) и механизм разрешения конфликтов синхронизации. |

### Допущения (Assumptions)

| № | Допущение | Описание | Риск, если допущение нарушится |
| :---: | :--- | :--- | :--- |
| **A-01** | **Платёжеспособная аудитория** | Пользователи приложения имеют средний и выше среднего доход и готовы покупать товары премиум-сегмента. | Если основная аудитория окажется чувствительной к цене, конверсия в покупки будет ниже целевых 18%. Потребуется пересмотр бизнес-модели. |
| **A-02** | **Доступность сторонних API** | Все внешние сервисы (погода, карты, носимые устройства) имеют стабильные и документированные API с SLA ≥ 99.9%. | Падение внешнего сервиса делает функцию недоступной. Требуется кеширование и fallback-механизмы (например, использование погоды за последний известный период). |
| **A-03** | **Принятие геймификации** | Целевая аудитория позитивно воспринимает игровые механики (ачивки, рейтинги). | Если пользователи сочтут геймификацию навязчивой или "детской", это может снизить вовлечённость вместо её повышения. Потребуется гибкая настройка: возможность отключить геймификацию. |
| **A-04** | **Масштабируемость ML-моделей** | Объём данных для обучения моделей будет достаточным уже на первом году (минимум 1 млн активных пользователей). | При медленном росте пользовательской базы качество рекомендаций и AI-коучинга будет низким, что ударит по цели №2. Возможно использование синтетических данных и transfer learning на старте. |
| **A-05** | **Юридическая простота B2B** | Партнёрские договоры с региональными фитнес-клубами могут быть заключены по единому шаблону с локальными дополнениями. | Если в каждой стране потребуется индивидуальный юридический договор с длительными согласованиями, цель №4 (500+ партнёров) становится недостижимой в срок. |
| **A-06** | **Лояльность пользователей к бренду** | Бренд компании уже узнаваем и пользуется доверием, что снижает порог установки приложения. | Если имидж бренда пострадает (скандалы, качество продукции), это напрямую скажется на установках и удержании, независимо от качества приложения. |
| **A-07** | **Техническая готовность пользователей** | Целевая аудитория имеет современные смартфоны (iOS 15+, Android 11+) и базовые навыки работы с приложениями. | Поддержка старых версий ОС увеличивает стоимость разработки и тестирования. Решение: поддержка только двух последних мажорных версий. |

### Взаимосвязь ограничений и допущений с архитектурными решениями

| Ограничение/Допущение | Архитектурное решение |
| :--- | :--- |
| C-01 (GDPR/CCPA) | Внедрение Data Mesh с политиками доступа на уровне строк (Row-Level Security). Хранение PII в отдельном сегменте с шифрованием. |
| C-02 (Multi-cloud) | Использование Kubernetes и операторов (Crossplane) для декларативного управления ресурсами во всех облаках. |
| C-03 (Legacy) | API Gateway (Kong/Envoy) с маршрутизацией и трансформацией запросов. Паттерн Anti-Corruption Layer (ACL). |
| C-05 (Сжатые сроки) | Feature Toggles для поэтапного включения функций. Использование BaaS (Firebase/Auth0) для ускорения аутентификации. |
| C-07 (Офлайн) | CRDT (Conflict-free Replicated Data Types) или паттерн "Event Sourcing + CQRS" для разрешения конфликтов синхронизации. |
| A-02 (Внешние API) | Реализация паттерна Circuit Breaker и Bulkhead для изоляции сбоев внешних сервисов. |
| A-06 (Лояльность к бренду) | Внедрение системы NPS и мониторинг отзывов в приложении с автоматическим алертингом при падении оценки. |
