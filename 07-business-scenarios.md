# №7: Критические бизнес-сценарии (Business Scenarios)

---

**Назначение раздела:** Описание критических бизнес-сценариев позволяет определить ключевые потоки использования системы, которые имеют наибольшую ценность для бизнеса и пользователей. Эти сценарии формируют основу для тестирования, приоритизации разработки и оценки архитектурных решений. Каждый сценарий описывает, какие действия выполняет пользователь, какие компоненты системы задействованы и как система должна реагировать.

---

## 1. Методология описания сценариев

Каждый сценарий описывается по следующей структуре:

| Параметр | Описание |
|:---|:---|
| **ID** | Уникальный идентификатор сценария |
| **Название** | Краткое название сценария |
| **Описание** | Что делает пользователь и как система реагирует |
| **Участники** | Кто задействован в сценарии (пользователь, партнёр, система) |
| **Предусловия** | Что должно быть выполнено до начала сценария |
| **Шаги** | Последовательность действий |
| **Постусловия** | Состояние системы после завершения сценария |
| **Альтернативные потоки** | Возможные отклонения от основного сценария |
| **Бизнес-цель** | Связанная бизнес-цель (БТ-1..БТ-4) |
| **Связанные ФТ** | Функциональные требования, задействованные в сценарии |
| **Критичность** | Высокая / Средняя / Низкая |

## 2. Сценарии по бизнес-целям

### 2.1. БТ-1: Удержание через социум

---

#### Сценарий БС-01: Создание и развитие социальной группы

| Параметр | Значение |
|:---|:---|
| **ID** | БС-01 |
| **Название** | Создание и развитие социальной группы |
| **Описание** | Пользователь создаёт социальную группу по интересам (вид спорта, локация, уровень подготовки). Группа самоорганизуется: появляются новые участники, модераторы, публикуются посты и обсуждения. Группа становится самоподдерживающимся сообществом. |
| **Участники** | Пользователь (создатель группы), участники (подписчики), модераторы |
| **Предусловия** | Пользователь зарегистрирован и аутентифицирован в системе |
| **Шаги** | 1. Пользователь заполняет форму создания группы (название, описание, категория, теги, локация, уровень подготовки).<br>2. Система создаёт группу и назначает пользователя администратором.<br>3. Система отображает группу в поиске и рекомендует её пользователям со схожими интересами.<br>4. Другие пользователи находят группу через поиск или рекомендации и отправляют запрос на вступление.<br>5. Администратор или модераторы одобряют заявки.<br>6. Участники публикуют посты, обсуждают темы, предлагают совместные тренировки.<br>7. Группа растёт: система автоматически предлагает стать модераторами активным участникам. |
| **Постусловия** | Группа создана, активна, имеет участников и развивается без активного вмешательства администраторов |
| **Альтернативные потоки** | 1. Группа может быть закрытой (требуется одобрение заявок) или открытой (свободное вступление).<br>2. При нарушении правил модератор может заблокировать участника.<br>3. При отсутствии активности группа может быть автоматически помечена как "неактивная". |
| **Бизнес-цель** | БТ-1 (Удержание через социум) |
| **Связанные ФТ** | ФТ-5 (Социальные группы) |
| **Критичность** | Высокая |

#### Сценарий БС-02: Участие в челлендже

| Параметр | Значение |
|:---|:---|
| **ID** | БС-02 |
| **Название** | Участие в челлендже |
| **Описание** | Пользователь участвует в командном или индивидуальном челлендже (например, "Пробеги 50 км за неделю" или "Командный марафон"). В процессе челленджа пользователь видит свой прогресс, прогресс команды и рейтинг участников. После завершения челленджа победители получают виртуальные награды (XP, бейджи, монеты). |
| **Участники** | Пользователь, участники челленджа (индивидуальные или команды), система |
| **Предусловия** | Пользователь зарегистрирован; челлендж активен; пользователь записал хотя бы одну тренировку |
| **Шаги** | 1. Пользователь просматривает список доступных челленджей (индивидуальные и командные).<br>2. Пользователь выбирает челлендж и присоединяется к нему (или создаёт команду).<br>3. Система отображает пользователю цель челленджа, прогресс, рейтинг участников.<br>4. Пользователь записывает тренировки (ФТ-1), система автоматически обновляет прогресс в челлендже.<br>5. Пользователь видит уведомления о прогрессе команды и достижениях других участников (ФТ-15).<br>6. По окончании челленджа система подводит итоги: определяет победителей, начисляет XP, бейджи, виртуальные монеты.<br>7. Система отправляет уведомления о результатах челленджа всем участникам. |
| **Постусловия** | Челлендж завершён; победители получили награды; участники получили опыт и мотивацию |
| **Альтернативные потоки** | 1. Пользователь может покинуть челлендж до его завершения (тогда награды не начисляются).<br>2. При недостаточной активности участник может быть исключён из челленджа. |
| **Бизнес-цель** | БТ-1 (Удержание через социум) |
| **Связанные ФТ** | ФТ-6 (Геймификация), ФТ-1 (Запись тренировок) |
| **Критичность** | Высокая |

### 2.2. БТ-2: AI-коучинг

---

#### Сценарий БС-03: Получение персонализированного тренировочного плана

| Параметр | Значение |
|:---|:---|
| **ID** | БС-03 |
| **Название** | Получение персонализированного тренировочного плана |
| **Описание** | Пользователь получает автоматически сгенерированный тренировочный план на неделю/месяц на основе своих целей, уровня подготовки, инвентаря и истории тренировок. План включает рекомендации по нагрузке, типам тренировок и дням отдыха. |
| **Участники** | Пользователь, система (AI/ML-сервис) |
| **Предусловия** | Пользователь зарегистрирован; заполнил профиль (цели, уровень, возраст, вес); добавил инвентарь (ФТ-7); записал хотя бы 3–5 тренировок (ФТ-1) |
| **Шаги** | 1. Пользователь открывает раздел "Тренировочные планы".<br>2. Система анализирует: цели пользователя, уровень подготовки, инвентарь, историю тренировок, данные с датчиков (пульс, усталость).<br>3. AI-сервис генерирует персонализированный план на выбранный период (неделя/месяц).<br>4. План включает: дни тренировок, типы тренировок (интервальные, длительные, восстановительные), интенсивность, продолжительность.<br>5. Система отображает план пользователю с возможностью редактирования (перенести тренировку, изменить нагрузку).<br>6. Пользователь утверждает план, система сохраняет его и напоминает о предстоящих тренировках. |
| **Постусловия** | План сгенерирован, сохранён, пользователь начал его выполнение |
| **Альтернативные потоки** | 1. Если данных недостаточно (менее 3 тренировок), система предлагает общий план на основе целей и уровня.<br>2. Пользователь может запросить перегенерацию плана с изменёнными параметрами. |
| **Бизнес-цель** | БТ-2 (AI-коучинг) |
| **Связанные ФТ** | ФТ-4 (AI-формирование плана), ФТ-1 (Запись тренировок), ФТ-7 (Инвентарь) |
| **Критичность** | Высокая |

#### Сценарий БС-04: Автоматическая корректировка плана на основе данных тренировки

| Параметр | Значение |
|:---|:---|
| **ID** | БС-04 |
| **Название** | Автоматическая корректировка плана на основе данных тренировки |
| **Описание** | После завершения тренировки система анализирует показатели (пульс, темп, усталость) и при необходимости корректирует предстоящие тренировки в плане. Например, если пользователь перетренировался, система заменяет интенсивную тренировку на восстановительную. |
| **Участники** | Пользователь, система (AI/ML-сервис) |
| **Предусловия** | Пользователь выполняет тренировочный план (БС-03); завершил тренировку (ФТ-1) с подключённым датчиком пульса (ФТ-2) |
| **Шаги** | 1. Пользователь завершает тренировку, система сохраняет данные (ФТ-1).<br>2. AI-сервис анализирует данные: пульсовые зоны, вариабельность сердечного ритма (HRV), темп, субъективную оценку пользователя (если заполнена).<br>3. Система определяет уровень усталости и восстановления.<br>4. Если уровень усталости превышает пороговое значение, система корректирует план: заменяет следующую интенсивную тренировку на восстановительную, увеличивает время отдыха.<br>5. Пользователь получает уведомление об изменениях в плане. |
| **Постусловия** | План скорректирован с учётом текущего состояния пользователя |
| **Альтернативные потоки** | 1. Если данные с датчиков отсутствуют, система использует исторические данные и субъективную оценку.<br>2. Пользователь может отклонить предложенные корректировки. |
| **Бизнес-цель** | БТ-2 (AI-коучинг) |
| **Связанные ФТ** | ФТ-4 (AI-формирование плана), ФТ-1 (Запись тренировок), ФТ-2 (Датчики) |
| **Критичность** | Высокая |

### 2.3. БТ-3: Монетизация

---

#### Сценарий БС-05: Рекомендация и покупка нового инвентаря

| Параметр | Значение |
|:---|:---|
| **ID** | БС-05 |
| **Название** | Рекомендация и покупка нового инвентаря |
| **Описание** | Система отслеживает пробег обуви пользователя. При достижении критического значения (например, 800 км) система отправляет уведомление о необходимости замены и рекомендует новые модели с учётом стиля бега пользователя. Пользователь переходит в магазин и оформляет покупку. |
| **Участники** | Пользователь, система, legacy e-commerce |
| **Предусловия** | Пользователь добавил обувь в инвентарь (ФТ-7); пробег превысил критическое значение; пользователь зарегистрирован |
| **Шаги** | 1. Система автоматически отслеживает пробег обуви на основе записей тренировок (ФТ-1).<br>2. При достижении критического значения (например, 800 км) система отправляет push-уведомление и in-app уведомление (ФТ-23).<br>3. В уведомлении система указывает: текущий пробег, рекомендованные модели с учётом стиля бега (пронация, тип поверхности).<br>4. Пользователь нажимает на уведомление или открывает раздел "Инвентарь".<br>5. Система отображает карточки рекомендованных товаров с ценой, описанием и кнопкой "Купить".<br>6. Пользователь нажимает "Купить", система бесшовно переводит его в интернет-магазин (единый логин, единая корзина) (ФТ-8).<br>7. Пользователь оформляет заказ и оплачивает (интеграция с платежной системой).<br>8. Система получает подтверждение о покупке и автоматически добавляет новый инвентарь в профиль пользователя (или предлагает добавить). |
| **Постусловия** | Покупка совершена; новый инвентарь добавлен в профиль пользователя |
| **Альтернативные потоки** | 1. Пользователь может отклонить рекомендацию (тогда система отправит напоминание через 1–2 недели).<br>2. Если у пользователя отключены уведомления, рекомендация отображается в разделе "Инвентарь". |
| **Бизнес-цель** | БТ-3 (Монетизация) |
| **Связанные ФТ** | ФТ-7 (Инвентарь), ФТ-8 (Рекомендации), ФТ-1 (Запись тренировок) |
| **Критичность** | Высокая |

|:---|:---|
| **ID** | БС-06 |
| **Название** | Оформление премиум-подписки |
| **Описание** | Пользователь оформляет платную подписку на премиум-функции (AI-коучинг, расширенная статистика, продвинутые тренировочные планы). Подписка автоматически продлевается ежемесячно или ежегодно. |
| **Участники** | Пользователь, система, платежная система (Stripe / Apple Pay / Google Pay) |

| **Шаги** | 1. Пользователь открывает раздел "Подписка" или нажимает на заблокированную премиум-функцию.<br>2. Система отображает тарифы (ежемесячный, годовой) и список премиум-возможностей.<br>3. Пользователь выбирает тариф и нажимает "Оформить подписку".<br>4. Система перенаправляет пользователя на страницу оплаты (Stripe / Apple Pay / Google Pay).<br>5. Пользователь подтверждает платёж.<br>6. Система получает подтверждение от платёжной системы и активирует подписку.<br>7. Пользователь получает доступ ко всем премиум-функциям. |
| **Постусловия** | Подписка активна; премиум-функции доступны пользователю |
| **Альтернативные потоки** | 1. Платёж отклонён — система отображает ошибку и предлагает повторить попытку или использовать другой способ оплаты.<br>2. Пользователь может отменить подписку в любой момент (доступ сохраняется до окончания оплаченного периода). |
| **Бизнес-цель** | БТ-3 (Монетизация) |
| **Связанные ФТ** | ФТ-4 (AI-формирование плана), ФТ-8 (Рекомендации) |
| **Критичность** | Высокая |

### 2.4. БТ-4: B2B-платформа

#### Сценарий БС-07: Создание региональной промоакции партнёром

| Параметр | Значение |
|:---|:---|
| **ID** | БС-07 |
| **Название** | Создание региональной промоакции партнёром |
| **Описание** | Региональный партнёр (фитнес-клуб, спортивный магазин) через админ-панель создаёт промоакцию для пользователей в своём регионе. Акция отображается только у пользователей, которые находятся в этом регионе и соответствуют целевой аудитории (вид спорта, уровень). |
| **Участники** | Партнёр, система (админ-панель), пользователи (целевая аудитория) |
| **Предусловия** | Партнёр зарегистрирован и аутентифицирован в системе (роль "Партнёр") |
| **Шаги** | 1. Партнёр входит в админ-панель (ФТ-10).<br>2. Партнёр заполняет форму создания промоакции: название, описание, условия (скидка), период действия, регион (геолокация).<br>3. Партнёр выбирает целевую аудиторию: вид спорта, уровень подготовки, возрастную группу (опционально).<br>4. Партнёр публикует акцию.<br>5. Система проверяет акцию на соответствие правилам и публикует.<br>6. Пользователи в указанном регионе видят акцию в ленте новостей (ФТ-9) или в разделе "Промоакции".<br>7. Партнёр отслеживает эффективность акции в реальном времени (количество просмотров, переходов, конверсий). |
| **Постусловия** | Акция создана, опубликована, видна целевой аудитории; партнёр получает аналитику |
| **Альтернативные потоки** | 1. Акция может быть отклонена модератором при нарушении правил.<br>2. Партнёр может отредактировать или завершить акцию досрочно. |
| **Бизнес-цель** | БТ-4 (B2B-платформа) |
| **Связанные ФТ** | ФТ-9 (Региональные промоакции), ФТ-10 (Админ-панель) |
| **Критичность** | Средняя (для MVP) / Высокая (в долгосрочной перспективе) |

#### Сценарий БС-08: Интеграция с календарём спортивных событий и регистрация

| Параметр | Значение |
|:---|:---|
| **ID** | БС-08 |
| **Название** | Интеграция с календарём спортивных событий и регистрация |
| **Описание** | Пользователь просматривает предстоящие забеги/соревнования в своём регионе, выбирает событие и регистрируется на него через приложение. Система интегрируется с внешней системой регистрации организатора. |
| **Участники** | Пользователь, система, организатор события (внешняя система) |
| **Предусловия** | Пользователь зарегистрирован; событие добавлено в календарь (организатором или администратором) |
| **Шаги** | 1. Пользователь открывает раздел "События" (ФТ-11).<br>2. Система отображает предстоящие спортивные события в регионе пользователя (и за его пределами по запросу).<br>3. Пользователь выбирает событие и нажимает "Зарегистрироваться".<br>4. Система перенаправляет пользователя на внешнюю систему регистрации (или открывает форму внутри приложения).<br>5. Пользователь заполняет данные и подтверждает регистрацию.<br>6. Система сохраняет информацию о регистрации и добавляет событие в календарь пользователя (Google Calendar / Apple Calendar).<br>7. Пользователь получает напоминания о событии за 7, 3, 1 день до старта. |
| **Постусловия** | Пользователь зарегистрирован на событие; событие добавлено в календарь |

Пользователь получает уведомление о результатах и изменениях в плане. |

| **Бизнес-цель** | БТ-1 (Удержание через социум) |
| **Связанные ФТ** | ФТ-12 (Поиск людей), ФТ-16 (Совместные тренировки), ФТ-1 (Запись тренировок) |
