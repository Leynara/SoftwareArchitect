# №15: Анализ рисков созданной архитектуры и компромиссов (Architecture Risks & Tradeoffs)

---

**Назначение раздела:** Анализ рисков, связанных с принятыми архитектурными решениями, и описание компромиссов, на которые пришлось пойти при проектировании системы. Раздел дополняет начальный анализ рисков (№5), фокусируясь на рисках, возникающих именно из-за архитектурных выборов: микросервисов, Event-Driven подхода, Polyglot Persistence, Cloud-agnostic стратегии и других решений. Для каждого риска указаны вероятность, влияние, стратегии mitigation и план действий.

---

## 1. Методология анализа

| Параметр | Описание |
|:---|:---|
| **Вероятность (P)** | Низкая (1–3) / Средняя (4–6) / Высокая (7–10) |
| **Влияние (I)** | Низкое (1–3) / Среднее (4–6) / Критическое (7–10) |
| **Приоритет** | Произведение вероятности и влияния (P × I) |
| **Связанные ADR** | Какие архитектурные решения создают этот риск |
| **Стратегия mitigation** | Действия для снижения вероятности или влияния |
| **План действий** | Что делать, если риск материализовался |

## 2. Риски, связанные с микросервисной архитектурой

### РА-01: Сложность управления распределённой системой

| Параметр | Значение |
|:---|:---|
| **Описание** | Микросервисная архитектура (ADR-001) требует управления множеством сервисов, их версиями, зависимостями, сетевой конфигурацией. Это увеличивает сложность эксплуатации, отладки и мониторинга. Команда может не справиться с операционной сложностью, что приведёт к частым сбоям и увеличению MTTR. |
| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Высокий (48) |
| **Связанные ADR** | ADR-001 (Микросервисы), ADR-008 (Cloud-agnostic), ADR-010 (Kubernetes) |
| **Стратегия mitigation** | 1. Внедрение Service Mesh (Istio/Linkerd) для управления трафиком.<br>2. Автоматизация развертывания (ArgoCD, GitOps).<br>3. Централизованное логирование и трассировка (ELK, Jaeger).<br>4. Обучение команды (Kubernetes, микросервисы).<br>5. Постепенное внедрение: сначала 3–4 сервиса, потом масштабирование. |
| **План действий** | 1. Усиление DevOps-команды.<br>2. Пересмотр границ сервисов (возможное объединение некоторых сервисов).<br>3. Внедрение дополнительных инструментов наблюдаемости. |
| **Компромисс** | Микросервисы дают гибкость и масштабируемость, но увеличивают операционную сложность. Сложность оправдана бизнес-требованиями роста до 50M пользователей. |

---

### РА-02: Сетевые задержки и увеличение времени отклика

| Параметр | Значение |
|:---|:---|
| **Описание** | В микросервисной архитектуре один запрос пользователя может проходить через несколько сервисов (API Gateway → BFF → Service A → Service B). Это увеличивает общее время отклика. Сетевые задержки между сервисами, особенно при Multi-cloud развертывании, могут превысить допустимые 200 мс. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Средний (30) |
| **Связанные ADR** | ADR-001 (Микросервисы), ADR-003 (API Gateway + BFF), ADR-008 (Multi-cloud) |
| **Стратегия mitigation** | 1. Кеширование (Redis) на уровне API Gateway и BFF.<br>2. Асинхронная обработка через Kafka для некритичных операций.<br>3. Размещение сервисов в одном регионе/облаке для минимизации задержек.<br>4. BFF агрегирует данные из нескольких сервисов, уменьшая число запросов.<br>5. Использование gRPC вместо REST для межсервисной коммуникации. |
| **План действий** | 1. Мониторинг времени ответа всех сервисов.<br>2. Оптимизация критических путей.<br>3. Пересмотр архитектуры BFF. |
| **Компромисс** | Масштабируемость и гибкость достигаются ценой потенциального увеличения задержки. Кеширование и асинхронность компенсируют этот эффект. |

---

### РА-03: Распределённые транзакции (Saga Pattern)

| Параметр | Значение |
|:---|:---|
| **Описание** | Бизнес-операции, требующие согласованности данных в нескольких сервисах (например, покупка инвентаря → обновление инвентаря → начисление XP), требуют реализации распределённых транзакций через Saga Pattern. Реализация Saga сложна, особенно компенсирующие операции при сбоях. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Критическое (7) |
| **Приоритет** | Высокий (42) |
| **Связанные ADR** | ADR-001 (Event-Driven), ADR-002 (Kafka) |
| **Стратегия mitigation** | 1. Использование Choreography-based Saga (события через Kafka) вместо Orchestration.<br>2. Тщательное проектирование компенсирующих операций.<br>3. Тестирование сценариев сбоев (Chaos Engineering).<br>4. Использование фреймворков для Saga (например, Temporal, Camunda).<br>5. Документирование всех Saga-потоков. |
| **План действий** | 1. Аудит всех Saga-потоков.<br>2. Улучшение мониторинга Saga (отслеживание состояния).<br>3. Внедрение ручных компенсирующих операций для сложных случаев. |
| **Компромисс** | Распределённые транзакции сложнее локальных, но позволяют сохранить автономность сервисов и масштабируемость. |

## 3. Риски, связанные с Event-Driven архитектурой

### РА-04: Eventual Consistency и проблемы с данными

| Параметр | Значение |
|:---|:---|
| **Описание** | Event-Driven архитектура (ADR-001, ADR-002) использует Eventual Consistency — данные в разных сервисах могут быть несогласованными в течение некоторого времени. Это может привести к ситуациям, когда пользователь видит устаревшие данные (например, XP не обновились сразу после тренировки). |
| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Высокий (40) |
| **Связанные ADR** | ADR-001 (Event Sourcing), ADR-002 (Kafka) |
| **Стратегия mitigation** | 1. Чёткое документирование, какие данные являются eventually consistent, а какие — строго согласованными.<br>2. Для критических данных (платежи) использовать строгую консистентность.<br>3. Визуализация статуса синхронизации для пользователя (например, "XP будут начислены в течение минуты").<br>4. Мониторинг задержек событий (Kafka lag).<br>5. Использование компенсирующих событий при ошибках. |
| **План действий** | 1. Увеличение приоритета обработки событий.<br>2. Пересмотр модели консистентности для критических операций.<br>3. Внедрение механизмов ручной синхронизации. |
| **Компромисс** | Eventual Consistency повышает производительность и масштабируемость, но требует управления ожиданиями пользователей и тщательного проектирования. |

---

### РА-05: Сложность отладки асинхронных потоков

| Параметр | Значение |
|:---|:---|
| **Описание** | Асинхронные потоки событий через Kafka (ADR-002) сложнее отлаживать, чем синхронные запросы. Ошибка может возникнуть в одном сервисе, а проявиться в другом через несколько минут. Это увеличивает время диагностики инцидентов. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Средний (36) |
| **Связанные ADR** | ADR-002 (Kafka), ADR-009 (Наблюдаемость) |
| **Стратегия mitigation** | 1. Сквозная трассировка (Jaeger) с корреляционными ID.<br>2. Логирование всех событий с timestamp и trace ID.<br>3. Дашборды для мониторинга Kafka-потоков (латентность, ошибки).<br>4. Тестирование асинхронных сценариев.<br>5. Использование Kafka Streams для обработки событий с возможностью replay. |
| **План действий** | 1. Улучшение инструментов наблюдаемости.<br>2. Обучение команды работе с асинхронными системами.<br>3. Внедрение дополнительных алертов на аномалии в потоках. |
| **Компромисс** | Асинхронность даёт масштабируемость и слабую связанность, но усложняет отладку. Наблюдаемость — ключевой элемент для управления этим риском. |

---

## 4. Риски, связанные с Polyglot Persistence

### РА-06: Сложность управления несколькими БД

| Параметр | Значение |
|:---|:---|
| **Описание** | Использование 6+ различных баз данных (PostgreSQL, TimescaleDB, Elasticsearch, Redis, Neo4j, MinIO, DWH) требует специалистов по каждой технологии, увеличивает затраты на поддержку и усложняет синхронизацию данных между БД. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Средний (30) |
| **Связанные ADR** | ADR-004 (Polyglot Persistence), ADR-006 (TimescaleDB) |
| **Стратегия mitigation** | 1. Чёткое разделение ответственности между БД.<br>2. Использование managed-сервисов в облаках (AWS RDS, Elasticsearch Service) для снижения операционной нагрузки.<br>3. Обучение команды (1–2 специалиста на каждую БД).<br>4. Документирование схем и запросов.<br>5. Использование единого слоя доступа к данным (Data Access Layer). |
| **План действий** | 1. Пересмотр необходимости каждой БД.<br>2. Консолидация некоторых БД (например, TimescaleDB как расширение PostgreSQL).<br>3. Привлечение внешних экспертов. |
| **Компромисс** | Polyglot Persistence даёт оптимальную производительность для каждого типа данных, но увеличивает сложность управления и требует широкого набора компетенций. |

---

### РА-07: Проблемы синхронизации между БД

| Параметр | Значение |
|:---|:---|
| **Описание** | Данные часто дублируются между БД для разных целей (например, профиль пользователя в PostgreSQL и Elasticsearch для поиска). Синхронизация между БД может иметь задержки или ошибки, приводя к несоответствию данных. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Средний (30) |
| **Связанные ADR** | ADR-004 (Polyglot Persistence), ADR-005 (Шардирование) |
| **Стратегия mitigation** | 1. Использование CDC (Change Data Capture) — например, Debezium для автоматической синхронизации.<br>2. Использование Event Sourcing для гарантированной синхронизации через события.<br>3. Мониторинг задержек синхронизации.<br>4. Периодическая проверка согласованности данных. |
| **План действий** | 1. Ручная синхронизация при обнаружении расхождений.<br>2. Автоматическое исправление несоответствий через фоновые процессы.<br>3. Пересмотр архитектуры синхронизации. |
| **Компромисс** | Дублирование данных повышает производительность поиска и чтения, но требует управления синхронизацией. |

---

## 5. Риски, связанные с Multi-cloud стратегией

### РА-08: Увеличение сложности инфраструктуры

| Параметр | Значение |
|:---|:---|
| **Описание** | Cloud-agnostic подход (ADR-008) с использованием Kubernetes и открытых стандартов требует управления несколькими облаками (AWS, GCP, Azure), что увеличивает сложность инфраструктуры, стоимость и время настройки. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Средний (36) |
| **Связанные ADR** | ADR-008 (Cloud-agnostic), ADR-010 (Kubernetes) |
| **Стратегия mitigation** | 1. Начало с одного облака, постепенный переход к Multi-cloud.<br>2. Использование Terraform для унифицированного управления ресурсами.<br>3. Стандартизация инструментов (Kubernetes, PostgreSQL, Kafka) во всех облаках.<br>4. Обучение DevOps-команды работе с несколькими облаками.<br>5. Автоматизация развертывания через GitOps (ArgoCD). |
| **План действий** | 1. Усиление DevOps-команды.<br>2. Пересмотр стратегии: возможно, остаться в 2 облаках вместо 3.<br>3. Использование абстрактных интерфейсов для облачных сервисов. |
| **Компромисс** | Multi-cloud даёт гибкость и отсутствие vendor lock-in, но требует больших усилий на управление. Гибкость оправдана бизнес-стратегией. |

## 6. Риски, связанные с AI / ML

### РА-09: Качество ML-моделей ниже ожидаемого

| Параметр | Значение |
|:---|:---|
| **Описание** | ML-модели для генерации тренировочных планов могут давать рекомендации низкого качества из-за недостатка данных, неправильного выбора алгоритмов или концептуальных ошибок. Это может снизить доверие пользователей к AI-коучингу. |
| **Вероятность** | Средняя (6) |
| **Влияние** | Критическое (8) |
| **Приоритет** | Высокий (48) |
| **Связанные ADR** | ADR-007 (Rule-based → ML) |
| **Стратегия mitigation** | 1. На MVP — rule-based система, сбор данных для обучения.<br>2. Transfer learning для ускорения разработки.<br>3. A/B-тестирование различных ML-подходов.<br>4. Привлечение спортивных физиологов для валидации.<br>5. Постоянный мониторинг качества рекомендаций (offline-метрики). |
| **План действий** | 1. Откат на rule-based версию.<br>2. Досбор данных и переобучение моделей.<br>3. Пересмотр метрик качества. |
| **Компромисс** | Rule-based на MVP обеспечивает быстрый запуск, но ML-модели требуют времени и данных для достижения качества. |

## 7. Риски, связанные с масштабированием

### РА-10: Шардирование БД может не справиться с ростом

| Параметр | Значение |
|:---|:---|
| **Описание** | Шардирование по пользователю и региону (ADR-005) может оказаться недостаточно эффективным при росте до 50M пользователей. Кросс-шардовые запросы могут стать узким местом, особенно для глобальных рейтингов и аналитики. |
| **Вероятность** | Низкая (4) |
| **Влияние** | Критическое (8) |
| **Приоритет** | Высокий (32) |
| **Связанные ADR** | ADR-005 (Комбинированное шардирование) |
| **Стратегия mitigation** | 1. Использование DWH/BigQuery для аналитических запросов (не на OLTP).<br>2. Агрегация данных в реальном времени через Kafka Streams.<br>3. Использование Elasticsearch для глобального поиска (не требует шардирования).<br>4. Мониторинг нагрузки на шарды и проактивное добавление новых шардов. |
| **План действий** | 1. Решардинг (добавление новых шардов).<br>2. Миграция данных на новые шарды.<br>3. Пересмотр модели данных. |
| **Компромисс** | Шардирование позволяет масштабироваться, но усложняет аналитические запросы. DWH и Elasticsearch компенсируют это. |

---

### РА-11: Высокая стоимость инфраструктуры при масштабировании

| Параметр | Значение |
|:---|:---|
| **Описание** | Хранение 5M тренировок в день и обслуживание 50M пользователей может привести к неконтролируемому росту затрат на инфраструктуру (cloud costs). Особенно критично для TimescaleDB, S3-хранилища и Kafka-кластеров. |
| **Вероятность** | Высокая (8) |
| **Влияние** | Среднее (6) |
| **Приоритет** | Высокий (48) |
| **Связанные ADR** | ADR-004 (Polyglot Persistence), ADR-006 (TimescaleDB), ADR-008 (Multi-cloud) |
| **Стратегия mitigation** | 1. Использование сжатия данных в TimescaleDB (до 90%).<br>2. Автоматическое архивирование старых данных в холодное хранилище (Glacier).<br>3. Использование spot-инстансов для некритичных вычислений.<br>4. Мониторинг затрат в реальном времени (Cloud Cost Management).<br>5. Правильная политика TTL для логов и событий. |
| **План действий** | 1. Оптимизация запросов и данных.<br>2. Пересмотр политики хранения.<br>3. Смена провайдера для некоторых сервисов (например, использование более дешёвого S3-подобного хранилища). |
| **Компромисс** | Хранение всех данных обеспечивает качество аналитики и ML, но требует затрат. Баланс достигается через TTL и архивирование. |

## 8. Риски, связанные с безопасностью

### РА-12: Уязвимости в распределённой системе

| Параметр | Значение |
|:---|:---|
| **Описание** | Микросервисная архитектура увеличивает поверхность атаки: каждый сервис, API-эндпоинт и сетевой порт — потенциальная точка входа для злоумышленников. Также возрастает риск неправильной настройки безопасности (например, открытые порты, слабые пароли). |
| **Вероятность** | Низкая (3) |
| **Влияние** | Критическое (10) |
| **Приоритет** | Высокий (30) |
| **Связанные ADR** | ADR-001 (Микросервисы), ADR-011 (Безопасность) |
| **Стратегия mitigation** | 1. Регулярные пентесты (не реже 1 раза в квартал).<br>2. Использование Service Mesh (Istio) для управления безопасностью между сервисами (mTLS).<br>3. Внедрение WAF (Web Application Firewall) на уровне API Gateway.<br>4. Обучение команды DevSecOps практикам.<br>5. Использование Infrastructure as Code для консистентной настройки безопасности. |
| **План действий** | 1. Немедленное исправление уязвимостей.<br>2. Усиление мониторинга безопасности.<br>3. Внешний аудит безопасности. |
| **Компромисс** | Микросервисы дают гибкость, но увеличивают поверхность атаки. Регулярные проверки и Service Mesh минимизируют этот риск. |

## 9. Риски, связанные с интеграциями

### РА-13: Зависимость от внешних API (карты, погода, устройства)

| Параметр | Значение |
|:---|:---|
| **Описание** | Система зависит от внешних API для карт, погоды, импорта данных из устройств. Сбой внешнего API может сделать соответствующие функции недоступными (например, отображение маршрута). |
| **Вероятность** | Средняя (5) |
| **Влияние** | Среднее (5) |
| **Приоритет** | Средний (25) |
| **Связанные ADR** | ADR-002 (Гибридная коммуникация) |
| **Стратегия mitigation** | 1. Кеширование внешних данных (погода — 1 час, карты — офлайн-кеш).<br>2. Fallback-механизмы (например, использование OpenStreetMap вместо Google Maps).<br>3. Circuit Breaker для изоляции сбоев внешних API.<br>4. Мониторинг доступности внешних API. |
| **План действий** | 1. Переключение на резервный API.<br>2. Уведомление пользователей о временной недоступности функции.<br>3. Усиление кеширования. |
| **Компромисс** | Интеграции расширяют функциональность, но создают зависимости. Кеширование и fallback снижают влияние сбоев. |

### РА-14: Интеграция с legacy e-commerce

| Параметр | Значение |
|:---|:---|
| **Описание** | Legacy e-commerce (SOAP, REST v1) может иметь нестабильную работу, ограниченную пропускную способность или несовместимые модели данных. Это может нарушить ключевую бизнес-функцию — бесшовные покупки. |
| **Вероятность** | Высокая (8) |
| **Влияние** | Критическое (8) |
| **Приоритет** | Высокий (64) |
| **Связанные ADR** | ADR-003 (API Gateway + Anti-Corruption Layer) |
| **Стратегия mitigation** | 1. Anti-Corruption Layer для трансформации и изоляции.<br>2. Кеширование данных из legacy (например, каталог товаров).<br>3. Поэтапная интеграция: сначала базовые операции, потом сложные.<br>4. Мониторинг доступности и производительности legacy-систем.<br>5. План замены или модернизации legacy (в долгосрочной перспективе). |
| **План действий** | 1. Увеличение времени ожидания (timeout) для legacy.<br>2. Ручная обработка критических случаев.<br>3. Ускорение модернизации legacy-системы. |
| **Компромисс** | Адаптация legacy сохраняет существующий бизнес, но требует дополнительных усилий и создаёт риски. Минимизация зависимости через ACL — приоритет. |

## 12. Компромиссы, принятые в архитектуре

| Компромисс | Решение | Причина | Альтернативы |
|:---|:---|:---|:---|
| **Сложность vs Гибкость** | Микросервисы вместо монолита | Требование независимой разработки команд; рост до 50M пользователей | Монолит был бы проще, но не масштабировался бы |
| **Консистентность vs Производительность** | Eventual Consistency | Глобальное распределение; пиковые нагрузки (марафоны) | Строгая консистентность увеличила бы задержки |
| **Multi-cloud vs Простота** | Cloud-agnostic подход | Бизнес-требование отсутствия vendor lock-in | Single-cloud был бы проще, но противоречил стратегии |
| **Стоимость vs Эффективность** | Open-source инструменты | Снижение затрат при сохранении качества | Коммерческие решения (Datadog, New Relic) дороже |
| **Офлайн vs Сложность** | Event Sourcing + CRDT | Критичное требование пользователей | Без офлайн-режима было бы проще, но потеряли бы пользователей |
| **AI-качество vs Скорость** | Rule-based MVP → ML | Быстрый запуск, сбор данных для обучения | Сразу ML заняло бы больше времени, риск провала |
