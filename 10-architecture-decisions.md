# №10: Архитектурные опции и выбор (Architecture Decisions)

---

**Назначение раздела:** Анализ и описание архитектурных опций, рассмотренных при проектировании системы, и обоснование выбранных решений. Раздел демонстрирует, что архитектурные решения приняты осознанно, с учётом бизнес-целей, функциональных и нефункциональных требований, а также ограничений и допущений. Для каждой опции указаны преимущества, недостатки и причины выбора.

---

## 1. Методология выбора архитектурных решений

Каждое архитектурное решение рассматривается по следующей структуре:

| Параметр | Описание |
|:---|:---|
| **Решение** | Краткое название принятого решения |
| **Контекст** | Какая проблема или требование требует решения |
| **Рассмотренные опции** | Альтернативные варианты с их преимуществами и недостатками |
| **Выбранное решение** | Принятый вариант и его обоснование |
| **Обоснование** | Почему выбран именно этот вариант (связь с бизнес-целями и НФТ) |
| **Компромиссы** | Чем пришлось пожертвовать |
| **Альтернативы, отклонённые** | Почему другие варианты не подошли |

## 2. Решения по архитектурному стилю

### Решение АС-01: Выбор архитектурного стиля

| Параметр | Значение |
|:---|:---|
| **Решение** | Микросервисная архитектура + Event-Driven (CQRS/Event Sourcing) |
| **Контекст** | Система должна масштабироваться до 50M пользователей, поддерживать независимую разработку команд, обеспечивать высокую доступность и отказоустойчивость. Компания уже использует облачные технологии и готова к распределённым системам. |
| **Рассмотренные опции** | 1. **Монолит** — единое приложение с одним кодом и БД.<br>2. **Модульный монолит** — монолит с чёткими модулями, но единым развертыванием.<br>3. **Микросервисы** — множество независимых сервисов с собственными БД и CI/CD.<br>4. **Serverless / FaaS** — функции как сервис (AWS Lambda). |
| **Выбранное решение** | Микросервисная архитектура + Event-Driven (CQRS/Event Sourcing) |
| **Обоснование** | 1. Независимость команд: 7+ команд могут работать параллельно.<br>2. Масштабируемость: каждый сервис масштабируется отдельно.<br>3. Отказоустойчивость: сбой в одном сервисе не валит всю систему.<br>4. Multi-cloud: сервисы можно размещать в разных облаках.<br>5. Event-Driven позволяет асинхронно обрабатывать пиковые нагрузки (марафоны).<br>6. CQRS/Event Sourcing обеспечивает аудит, восстановление и разделение нагрузки на чтение/запись. |
| **Компромиссы** | 1. Сложность разработки и отладки распределённой системы.<br>2. Увеличение затрат на инфраструктуру.<br>3. Eventual Consistency вместо строгой консистентности.<br>4. Необходимость управления распределёнными транзакциями (Saga). |
| **Альтернативы, отклонённые** | 1. **Монолит** — не позволит масштабировать отдельные компоненты; сложно поддерживать 50M пользователей.<br>2. **Serverless** — не подходит для долгих операций (аналитика); ограничения по времени выполнения. |

## 3. Решения по API и коммуникации

### Решение АС-02: Стиль взаимодействия между сервисами

| Параметр | Значение |
|:---|:---|
| **Решение** | Асинхронная коммуникация через Message Broker (Kafka) + синхронная (REST/gRPC) для критических операций |
| **Контекст** | Сервисы должны обмениваться данными, сохраняя слабую связанность. Некоторые операции требуют немедленного ответа (покупка, регистрация), другие могут быть асинхронными (обновление ленты, аналитика). |
| **Рассмотренные опции** | 1. **Синхронный REST** — все запросы синхронные, HTTP/JSON.<br>2. **Синхронный gRPC** — высокопроизводительный бинарный протокол.<br>3. **Асинхронный (Message Broker)** — RabbitMQ / Kafka.<br>4. **Гибридный** — синхронный для запросов, асинхронный для событий. |
| **Выбранное решение** | Гибридный: REST/gRPC для синхронных запросов, Kafka для событий |
| **Обоснование** | 1. Kafka обеспечивает гарантированную доставку, репликацию и replay событий.<br>2. Асинхронность разгружает критические пути (запись тренировок).<br>3. REST/gRPC используются для операций, где нужен немедленный ответ (проверка платежа).<br>4. Слабая связанность сервисов — изменения в одном сервисе не ломают другие. |
| **Компромиссы** | 1. Увеличение задержки для асинхронных операций.<br>2. Сложность отладки асинхронных потоков.<br>3. Необходимость управления eventual consistency. |
| **Альтернативы, отклонённые** | 1. **Только синхронный REST** — создаёт жёсткую связанность; каскадные отказы; плохая масштабируемость.<br>2. **Только асинхронный** — не подходит для операций, требующих немедленного ответа. |

### Решение АС-03: Клиент-серверное взаимодействие

| Параметр | Значение |
|:---|:---|
| **Решение** | API Gateway + BFF (Backend for Frontend) |
| **Контекст** | Мобильные и веб-клиенты имеют разные требования к данным. Мобильные устройства требуют минимального количества запросов и агрегированных данных. Веб-админка требует детальной информации. |
| **Рассмотренные опции** | 1. **Прямой вызов микросервисов** — клиент знает о всех сервисах.<br>2. **API Gateway** — единая точка входа с маршрутизацией.<br>3. **API Gateway + BFF** — отдельные BFF для разных клиентов (мобильный, веб).<br>4. **GraphQL Gateway** — единый endpoint с гибкими запросами. |
| **Выбранное решение** | API Gateway + BFF (для мобильных приложений) |
| **Обоснование** | 1. API Gateway обеспечивает единую точку входа, аутентификацию, rate limiting, кеширование.<br>2. BFF агрегирует данные из нескольких сервисов, уменьшая количество запросов с мобильного устройства.<br>3. Разные BFF для iOS/Android позволяют оптимизировать под платформу.<br>4. Упрощает клиентскую разработку. |
| **Компромиссы** | 1. Дополнительный слой увеличивает задержку.<br>2. BFF требует поддержки и синхронизации с клиентскими командами.<br>3. Возможен дублирование логики в BFF. |
| **Альтернативы, отклонённые** | 1. **Прямой вызов** — клиент становится сложным; много запросов с мобильного устройства; проблема с безопасностью.<br>2. **GraphQL** — сложность кеширования и ограничения запросов; требует дополнительного слоя. |

## 4. Решения по хранению данных

### Решение АС-04: Выбор типов баз данных (Polyglot Persistence)

| Параметр | Значение |
|:---|:---|
| **Решение** | Polyglot Persistence: PostgreSQL (OLTP) + TimescaleDB (Time-series) + Elasticsearch (Поиск) + Redis (Кеш) + Neo4j (Графовые) + MinIO/S3 (Объектное) + DWH/BigQuery (Аналитика) |
| **Контекст** | Система хранит разнородные данные: профили пользователей (транзакционные), тренировки (time-series), социальные связи (графовые), медиа (объектные), логи для поиска (текстовые). Одна БД не может эффективно обрабатывать все типы данных. |
| **Рассмотренные опции** | 1. **Одна БД (PostgreSQL)** — всё в одной реляционной БД.<br>2. **Две БД** — реляционная + time-series.<br>3. **Polyglot Persistence** — несколько специализированных БД. |
| **Выбранное решение** | Polyglot Persistence (множество специализированных БД) |
| **Обоснование** | 1. PostgreSQL: ACID для транзакционных данных (профили, заказы, подписки).<br>2. TimescaleDB: эффективное хранение и агрегация time-series данных (тренировки, пульс).<br>3. Elasticsearch: быстрый полнотекстовый и гео-поиск.<br>4. Redis: кеширование и управление сессиями с низкой задержкой.<br>5. Neo4j: эффективные графовые запросы для социальных связей ("друзья друзей").<br>6. MinIO/S3: объектное хранение для медиа, масштабируемое и дешёвое.<br>7. DWH/BigQuery: аналитика и ML-данные. |
| **Компромиссы** | 1. Сложность управления несколькими БД.<br>2. Необходимость синхронизации между БД.<br>3. Увеличение затрат на инфраструктуру.<br>4. Необходимость специалистов по разным БД. |
| **Альтернативы, отклонённые** | 1. **Одна БД** — не сможет эффективно обрабатывать time-series (миллионы записей) и графовые запросы одновременно; производительность упадёт.<br>2. **Только реляционная + time-series** — не поддерживает эффективный поиск и графовые связи. |

### Решение АС-05: Стратегия шардирования данных

| Параметр | Значение |
|:---|:---|
| **Решение** | Шардирование по пользователю (User-based sharding) + географическое шардирование |
| **Контекст** | При росте до 50M пользователей одна БД не сможет хранить все данные. Необходимо распределение данных по узлам. |
| **Рассмотренные опции** | 1. **Вертикальное шардирование** — разные таблицы на разных серверах.<br>2. **Горизонтальное шардирование** — строки одной таблицы на разных серверах (по пользователю).<br>3. **Географическое шардирование** — данные пользователей хранятся в их регионе.<br>4. **Комбинированное** — пользовательское + географическое. |
| **Выбранное решение** | Комбинированное: пользовательское шардирование + географическое |
| **Обоснование** | 1. Все данные пользователя находятся на одном шарде (простота запросов).<br>2. Географическое шардирование обеспечивает низкую задержку (данные рядом с пользователем).<br>3. Соответствие GDPR/CCPA (данные не покидают регион пользователя).<br>4. Лёгкость масштабирования — добавление новых шардов. |
| **Компромиссы** | 1. Сложность кросс-шардовых запросов (например, глобальные рейтинги).<br>2. Неравномерное распределение нагрузки.<br>3. Сложность миграции данных между шардами. |
| **Альтернативы, отклонённые** | 1. **Только пользовательское** — не учитывает юридические ограничения (GDPR).<br>2. **Только географическое** — сложно масштабировать при росте пользователей в регионе. |

## 5. Решения по обработке данных

### Решение АС-06: Обработка данных тренировок (Time-series)

| Параметр | Значение |
|:---|:---|
| **Решение** | TimescaleDB для хранения, Apache Kafka для потоковой обработки |
| **Контекст** | Система получает до 5M тренировок в день. Каждая тренировка содержит тысячи точек данных (GPS, пульс). Необходимо эффективно хранить, агрегировать и анализировать эти данные. |
| **Рассмотренные опции** | 1. **PostgreSQL с партиционированием**.<br>2. **TimescaleDB** (расширение PostgreSQL для time-series).<br>3. **InfluxDB** — специализированная time-series БД.<br>4. **Cassandra** — распределённая NoSQL. |
| **Выбранное решение** | TimescaleDB + Apache Kafka |
| **Обоснование** | 1. TimescaleDB: гибрид реляционной и time-series; поддержка SQL; автоматическое партиционирование по времени; сжатие до 90%.<br>2. Kafka: буферизация и потоковая обработка данных с устройств.<br>3. Совместимость с PostgreSQL (единый стек для команд).<br>4. Горизонтальное масштабирование через шардирование. |
| **Компромиссы** | 1. TimescaleDB требует тюнинга под high-load.<br>2. Kafka добавляет сложность в инфраструктуре.<br>3. Задержка между записью в Kafka и появлением в БД. |
| **Альтернативы, отклонённые** | 1. **PostgreSQL с партиционированием** — плохая производительность для time-series агрегации; нет встроенного сжатия.<br>2. **InfluxDB** — сложность интеграции с существующими SQL-запросами; ограниченный язык запросов.<br>3. **Cassandra** — сложность написания аналитических запросов; нет встроенных time-series функций. |

### Решение АС-07: AI / ML архитектура

| Параметр | Значение |
|:---|:---|
| **Решение** | Двухэтапная: Rule-based на MVP, ML-модели на этапе 2; отдельный ML-сервис с MLOps |
| **Контекст** | Система должна генерировать персонализированные тренировочные планы. На старте данных мало для качественного обучения ML. Необходимо быстро запустить MVP, а затем улучшать качество рекомендаций. |
| **Рассмотренные опции** | 1. **Rule-based система** — на основе экспертных знаний.<br>2. **ML-модели с нуля** — сбор данных, обучение, внедрение.<br>3. **Transfer learning** — адаптация существующих моделей.<br>4. **Двухэтапный подход** — сначала правила, потом ML. |
| **Выбранное решение** | Двухэтапный: Rule-based на MVP → ML на этапе 2 |
| **Обоснование** | 1. Rule-based позволяет запустить MVP без данных для обучения.<br>2. Сбор данных с первых пользователей для последующего обучения.<br>3. Transfer learning ускоряет разработку ML-моделей.<br>4. Отдельный ML-сервис позволяет независимо масштабировать и обновлять модели.<br>5. MLOps (MLflow / Kubeflow) для управления моделями. |
| **Компромиссы** | 1. Rule-based хуже персонализирует, чем ML.<br>2. Задержка между сбором данных и внедрением ML.<br>3. Необходимость поддерживать две версии рекомендаций. |
| **Альтернативы, отклонённые** | 1. **Сразу ML** — не хватит данных; низкое качество моделей; риск провала.<br>2. **Только rule-based** — не даст конкурентного преимущества; низкое качество персонализации. |

## 6. Решения по инфраструктуре

### Решение АС-08: Multi-cloud стратегия

| Параметр | Значение |
|:---|:---|
| **Решение** | Kubernetes + Terraform + открытые стандарты (cloud-agnostic) |
| **Контекст** | Компания не привязана к одному облачному провайдеру. 90% систем уже в облаках (AWS, GCP, Azure). Приложение должно работать в любом из них. |
| **Рассмотренные опции** | 1. **Single-cloud (AWS)** — все ресурсы в AWS.<br>2. **Single-cloud с резервом** — основной AWS, резерв GCP.<br>3. **Multi-cloud активный** — активное использование 2+ облаков.<br>4. **Cloud-agnostic** — абстракция от облаков через Kubernetes и открытые стандарты. |
| **Выбранное решение** | Cloud-agnostic (Kubernetes + открытые стандарты) |
| **Обоснование** | 1. Возможность выбора лучшего провайдера под каждую задачу (GCP для ML, AWS для вычислений).<br>2. Отсутствие vendor lock-in.<br>3. Использование Kubernetes как слоя абстракции.<br>4. Terraform + Crossplane для декларативного управления ресурсами во всех облаках.<br>5. Открытые стандарты: PostgreSQL, Kafka, Redis, Elasticsearch. |
| **Компромиссы** | 1. Сложность управления несколькими облаками.<br>2. Необходимость унификации инструментов мониторинга и CI/CD.<br>3. Ограниченный выбор облачных сервисов (только open-source). |
| **Альтернативы, отклонённые** | 1. **Single-cloud** — противоречит стратегии компании; риск vendor lock-in; невозможность выбора лучшего решения.<br>2. **Multi-cloud активный** — избыточная сложность; дорогое поддержание. |

### Решение АС-09: Выбор инструментов наблюдаемости

| Параметр | Значение |
|:---|:---|
| **Решение** | Prometheus + Grafana (метрики) + ELK Stack (логи) + Jaeger (трассировка) |
| **Контекст** | Распределённая микросервисная система требует эффективных инструментов для мониторинга, диагностики и отладки. |
| **Рассмотренные опции** | 1. **Datadog** — коммерческое решение (всё в одном).<br>2. **New Relic** — коммерческое.<br>3. **Open-source стек** — Prometheus + Grafana + ELK + Jaeger.<br>4. **AWS CloudWatch / GCP Operations** — встроенные облачные инструменты. |
| **Выбранное решение** | Open-source стек (Prometheus + Grafana + ELK + Jaeger) |
| **Обоснование** | 1. Низкая стоимость (open-source).<br>2. Гибкость и настраиваемость.<br>3. Отсутствие vendor lock-in (работает в любом облаке).<br>4. Сообщество и плагины.<br>5. Отличная интеграция с Kubernetes. |
| **Компромиссы** | 1. Больше усилий на настройку и поддержку.<br>2. Отсутствие готовых дашбордов (требуют разработки).<br>3. Сложность масштабирования ELK при большом объёме логов. |
| **Альтернативы, отклонённые** | 1. **Datadog / New Relic** — высокая стоимость при масштабировании; vendor lock-in.<br>2. **Cloud-native (CloudWatch)** — привязка к одному облаку; слабая интеграция с другими облаками. |

### Решение АС-10: Оркестрация и развертывание

| Параметр | Значение |
|:---|:---|
| **Решение** | Kubernetes + Helm + GitLab CI + ArgoCD |
| **Контекст** | Система состоит из множества микросервисов, которые должны развёртываться, масштабироваться и обновляться независимо. Требуется автоматизация CI/CD и управление конфигурациями. |
| **Рассмотренные опции** | 1. **Docker Compose** — для локальной разработки.<br>2. **Kubernetes + Helm** — полноценная оркестрация.<br>3. **Nomad + Consul** — альтернатива от HashiCorp.<br>4. **AWS ECS / GCP Cloud Run** — managed-решения. |
| **Выбранное решение** | Kubernetes + Helm + GitLab CI + ArgoCD |
| **Обоснование** | 1. Kubernetes — стандарт индустрии; работает во всех облаках.<br>2. Helm — управление чартами для сложных приложений.<br>3. GitLab CI — интеграция с репозиторием, автоматизация сборки.<br>4. ArgoCD — GitOps для декларативного управления состояниями.<br>5. Multi-cloud совместимость. |
| **Компромиссы** | 1. Сложность управления Kubernetes.<br>2. Высокая стоимость кластера.<br>3. Необходимость экспертизы Kubernetes. |
| **Альтернативы, отклонённые** | 1. **Docker Compose** — не подходит для production.<br>2. **AWS ECS / GCP Cloud Run** — привязка к облаку; ограниченная гибкость.<br>3. **Nomad** — меньшее сообщество и инструментов. |

## 7. Решения по безопасности

### Решение АС-11: Аутентификация и авторизация

| Параметр | Значение |
|:---|:---|
| **Решение** | OAuth 2.0 + OpenID Connect (Keycloak / Auth0) + RBAC |
| **Контекст** | Система должна обеспечивать безопасный доступ для разных групп пользователей (пользователи, модераторы, партнёры, администраторы). Требуется поддержка социального входа и MFA. |
| **Рассмотренные опции** | 1. **Самописная аутентификация** (JWT + Spring Security).<br>2. **Keycloak** (open-source).<br>3. **Auth0** (SaaS).<br>4. **AWS Cognito / GCP Identity Platform**. |
| **Выбранное решение** | Keycloak / Auth0 + OAuth 2.0 + OpenID Connect + RBAC |
| **Обоснование** | 1. OAuth 2.0 и OpenID — отраслевые стандарты.<br>2. Keycloak (open-source) — бесплатно; Auth0 (SaaS) — быстрее настройка.<br>3. RBAC обеспечивает гибкое управление правами.<br>4. Поддержка social login (Google, Facebook, Apple).<br>5. MFA опционально. |
| **Компромиссы** | 1. Keycloak требует настройки и поддержки.<br>2. Auth0 — платный при масштабировании.<br>3. Сложность кастомизации под бизнес-логику. |
| **Альтернативы, отклонённые** | 1. **Самописная** — высокий риск ошибок; slow time-to-market.<br>2. **AWS Cognito / GCP Identity** — vendor lock-in. |
