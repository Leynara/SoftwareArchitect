# 12. Сценарии использования приложения

---

## 12.1. UC-001. Регистрация и настройка профиля

### Актор
Новый пользователь

### Цель
Начать использовать приложение и получить персонализированный опыт.

### Предусловия
Пользователь установил приложение.

### Основной сценарий
1. Пользователь открывает приложение.
2. Выбирает способ регистрации.
3. Подтверждает email или внешний аккаунт.
4. Указывает базовые параметры: регион, виды спорта, уровень подготовки.
5. Настраивает приватность.
6. При желании добавляет обувь, инвентарь и устройства.
7. Система создаёт профиль.

### Альтернативные сценарии
- Пользователь пропускает часть настроек.
- Пользователь запрещает доступ к геолокации.
- Пользователь не подключает устройства.

### Архитектурные компоненты
- Mobile App
- API Gateway
- Identity Service
- User Profile Service
- Consent Service

### Критичные атрибуты качества
- безопасность;
- удобство;
- доступность.

**Критерии успеха:**
- ≥99.9% успешных регистраций  
- ≥99.9% успешных авторизаций  
- время ответа API (регистрация/логин) < 200 мс  
- ≥95% пользователей успешно завершают регистрацию  
- ≥95% пользователей успешно входят с первой попытки  
- корректная обработка ошибок (дубликаты аккаунтов, неверные данные, истёкшие токены)  
- отсутствие несанкционированного доступа

## 12.2. UC-002. Запись тренировки

### Актор
Пользователь

### Цель
Записать тренировку и получить результаты.

### Предусловия
Пользователь авторизован.

### Основной сценарий
1. Пользователь выбирает тип тренировки.
2. Приложение начинает запись.
3. Собираются данные GPS, времени, темпа, пульса и других датчиков.
4. После завершения тренировки данные отправляются на сервер.
5. Training Service сохраняет тренировку.
6. Система публикует событие `training.completed`.
7. Analytics Service обновляет показатели.
8. Пользователь видит результат.

### Альтернативные сценарии
- Пользователь был офлайн.
- Данные GPS частично потеряны.
- Устройство прислало неполные данные.
- Пользователь вручную редактирует тренировку.

### Архитектурные компоненты
- Mobile App
- Device Integration Service
- Training Service
- Event Bus
- Analytics Service

### Критичные атрибуты качества
- надёжность;
- масштабируемость;
- устойчивость к сетевым сбоям.

**Критерии успеха:**
- ≥99.9% тренировок успешно сохраняются  
- отсутствие потери данных  
- время записи < 300 мс  
- пользователь подтверждает корректность данных

## 12.3. UC-003. Сравнение прогресса

### Актор
Пользователь

### Цель
Понять, улучшаются ли результаты.

### Основной сценарий
1. Пользователь открывает экран прогресса.
2. Система загружает агрегированные показатели.
3. Пользователь видит сравнение с прошлой неделей, месяцем или личным рекордом.
4. Система показывает рекомендации.

### Архитектурные компоненты
- Mobile App
- API Gateway
- Analytics Service
- Recommendation Service
- Cache

### Критичные атрибуты качества
- производительность;
- актуальность данных;
- понятность визуализации.

**Критерии успеха:**
- время загрузки аналитики < 300 мс  
- ≥50% пользователей регулярно просматривают прогресс  
- отсутствие неконсистентных данных

## 12.4. UC-004. Поиск людей для совместной тренировки

### Актор
Пользователь

### Цель
Найти людей с похожими интересами или маршрутом.

### Основной сценарий
1. Пользователь открывает поиск.
2. Указывает вид спорта, район, маршрут или текущую активность.
3. Система ищет подходящих пользователей с учётом приватности.
4. Пользователь отправляет приглашение.
5. Получатель принимает или отклоняет приглашение.

### Архитектурные компоненты
- Search Service
- Social Graph Service
- User Profile Service
- Privacy Service
- Notification Service

### Риски
- раскрытие местоположения;
- нежелательные контакты;
- токсичное поведение.

### Меры
- настройки видимости;
- блокировки;
- жалобы;
- ограничение точности геолокации.

**Критерии успеха:**
- ≥30% пользователей взаимодействуют с другими  
- рост активности в ленте  
- увеличение DAU/MAU

Пользователь

### Цель

- Event Bus
- Leaderboard Store
- Notification Service

- масштабируемость;
- производительность;
- отказоустойчивость.

**Критерии успеха:**
- система выдерживает пиковую нагрузку  
- leaderboard обновляется < 200 мс  
- ≥95% пользователей успешно завершают участие

## 12.6. UC-006. Рекомендация обновить обувь

### Актор
Пользователь

### Цель
Получить своевременную рекомендацию по замене обуви.

### Основной сценарий
1. Пользователь добавляет обувь в профиль.
2. Каждая тренировка увеличивает пробег обуви.
3. Система сравнивает пробег с рекомендуемым лимитом.
4. Recommendation Service формирует совет.
5. Commerce Integration Service подбирает подходящие модели.
6. Пользователь переходит в магазин.

### Архитектурные компоненты
- User Profile Service
- Training Service
- Recommendation Service
- Commerce Integration Service

### Бизнес-ценность
- рост продаж;
- полезная рекомендация;
- снижение риска травм.

**Критерии успеха:**
- ≥20% пользователей взаимодействуют с рекомендациями  
- CTR рекомендаций растёт  
- положительное влияние на retention
