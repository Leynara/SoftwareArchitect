# №6: План поэтапной разработки и расширения системы (Development Plan)

---

**Назначение раздела:** План поэтапной разработки определяет очерёдность реализации основных бизнес-требований, распределение функциональности по релизным циклам, критически важные компоненты и зависимость этапов от архитектурных решений. План позволяет прогнозировать ресурсные затраты, координировать работы между командами и управлять рисками.

---

## 1. Общая стратегия разработки

| Принцип | Описание |
|:---|:---|
| **MVP First** | Запуск минимально жизнеспособного продукта через 9 месяцев с базовым набором функций для проверки гипотез и сбора обратной связи |
| **Итеративность** | Короткие циклы разработки (2-недельные спринты) с регулярными релизами |
| **Feature Toggles** | Поэтапное включение функций для пользователей (A/B-тестирование, канареечные релизы) |
| **API-first** | Разработка API перед фронтендом для параллельной работы команд |
| **Data-driven** | Сбор метрик и аналитики с первого дня для принятия решений |
| **Безопасность с первого дня** | Внедрение практик безопасной разработки (DevSecOps) с самого начала |

## 2. Этап 1: MVP (0–9 месяцев)

### 2.1. Цели этапа

- Запустить рабочий продукт на рынке.
- Проверить ключевые гипотезы (востребованность трекинга, социальных функций, AI-коучинга).
- Собрать первых пользователей и начать формирование сообщества.
- Получить первые данные для обучения ML-моделей.

### 2.3. Архитектурные компоненты (MVP)

| Компонент | Технология | Примечание |
|:---|:---|:---|
| **Мобильное приложение (iOS/Android)** | Swift / Kotlin | Нативная разработка, поддержка офлайн-режима (локальная БД — SQLite) |
| **API Gateway** | Kong / Envoy | Базовая маршрутизация, аутентификация (JWT) |
| **Сервис Users & Profiles** | Spring Boot / Python (FastAPI) | Регистрация, профили, инвентарь |
| **Сервис Tracking** | Spring Boot / Python (FastAPI) | Приём данных, базовая обработка, запись в БД |
| **Сервис Social (базовый)** | Spring Boot / Python (FastAPI) | Группы (создание, вступление), лента |
| **Сервис Gamification (базовый)** | Spring Boot / Python (FastAPI) | Уровни, XP, простые достижения |
| **Сервис AI (rule-based)** | Python (Flask/FastAPI) | Простые правила, без ML |
| **Сервис Commerce (базовый)** | Spring Boot / Python (FastAPI) | Интеграция с legacy e-commerce (базовая) |
| **PostgreSQL** | AWS RDS / GCP Cloud SQL | Профили, группы, инвентарь, заказы |
| **TimescaleDB** | Timescale Cloud / Self-managed | Тренировки, метрики (time-series) |
| **Redis** | Elasticache / Memorystore | Кеширование, сессии |
| **MinIO / S3** | Объектное хранилище | Аватары, медиа |
| **Message Broker** | RabbitMQ (базовый) | Асинхронные задачи (уведомления) |

### 2.4. Инфраструктура (MVP)

| Компонент | Технология | Примечание |
|:---|:---|:---|
| **Оркестрация** | Kubernetes (EKS / GKE) | Один кластер, один регион |
| **CI/CD** | GitLab CI | Автоматическая сборка, тестирование, деплой |
| **Мониторинг** | Prometheus + Grafana | Базовые метрики (CPU, память, ошибки) |
| **Логирование** | ELK Stack | Централизованное логирование |
| **Безопасность** | OAuth 2.0 (Auth0) | Базовая аутентификация, HTTPS |

### 2.5. Ключевые вехи (MVP)

| Веха | Срок | Результат |
|:---|:---:|:---|
| **Старт разработки** | Месяц 0 | Сформированы команды, утверждён план, настроена инфраструктура |
| **Прототип архитектуры** | Месяц 3 | Рабочий прототип с базовым трекингом и аутентификацией |
| **Внутреннее альфа-тестирование** | Месяц 6 | Стабильная версия для сотрудников компании (100+ пользователей) |
| **Закрытое бета-тестирование** | Месяц 8 | Приглашённые пользователи (1,000+), сбор обратной связи |
| **Релиз MVP** | Месяц 9 | Публичный релиз (App Store / Google Play) |

### 2.6. Критически важные компоненты MVP

| Компонент | Почему критичен | Риск при задержке |
|:---|:---|:---|
| **Сервис Tracking** | Ядро приложения; без трекинга все остальные функции бессмысленны | MVP не может быть запущен |
| **Мобильное приложение (iOS/Android)** | Основной интерфейс для пользователей | Нет доступа к рынку |
| **Офлайн-режим** | Ключевое требование для пользователей, тренирующихся без интернета | Потеря данных, негативный UX |
| **Аутентификация и безопасность** | Защита пользовательских данных с первого дня | Юридические риски, потеря доверия |
| **API Gateway** | Единая точка входа, маршрутизация | Без него клиенты не могут взаимодействовать с сервисами |

### 3.3. Архитектурные улучшения

| Компонент | Изменение | Причина |
|:---|:---|:---|
| **Сервис Tracking** | Внедрение CQRS + Event Sourcing | Масштабируемость, аудит, восстановление |
| **Сервис AI** | Переход от rule-based к ML | Повышение качества рекомендаций |
| **Message Broker** | Замена RabbitMQ на Apache Kafka | Большая пропускная способность, репликация событий |
| **Сервис B2B / Partners** | Новый сервис | Запуск B2B-платформы |
| **Neo4j (графовая БД)** | Добавление | Социальные связи, поиск людей |
| **Elasticsearch** | Добавление | Поиск пользователей, групп, событий |
| **PostgreSQL** | Добавление Read Replicas, шардирование | Рост нагрузки на чтение |
| **Kubernetes** | Добавление второго региона | Географическая доступность |

| Компонент | Почему критичен | Риск при задержке |
|:---|:---|:---|
| **Сервис AI (ML)** | Ключевое конкурентное преимущество | Потеря дифференциации, отток пользователей |
