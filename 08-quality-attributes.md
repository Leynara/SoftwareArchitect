# №8: Атрибуты качества (Quality Attributes)

---

**Назначение раздела:** Атрибуты качества определяют нефункциональные характеристики системы, которые являются критическими для успеха продукта. Они описывают, насколько хорошо система выполняет свои функции, а не что именно она делает. Атрибуты качества служат основой для архитектурных решений, выбора технологий и оценки соответствия системы бизнес-целям.

---

## 1. Методология определения атрибутов качества

Атрибуты качества выбраны на основе:
- Бизнес-целей (БТ-1..БТ-4)
- Функциональных требований (ФТ-1..ФТ-12)
- Анализа стейкхолдеров и их интересов
- Выявленных рисков реализации

Для каждого атрибута качества определены:
- **Описание** — что означает атрибут в контексте системы
- **Метрики** — измеримые показатели для оценки атрибута
- **Архитектурные решения** — как атрибут обеспечивается в архитектуре
- **Компромиссы** — какие решения приходится принимать для достижения атрибута
- **Связанные бизнес-цели и ФТ**

---

## 2. Основные атрибуты качества

По требованию задания выделено **три основных атрибута качества**, которые имеют приоритетное значение для системы.

### 2.1. Масштабируемость (Scalability)

| Параметр | Значение |
|:---|:---|
| **Описание** | Способность системы выдерживать рост количества пользователей, объёма данных и нагрузки без деградации производительности и с пропорциональным увеличением затрат на инфраструктуру. Система должна масштабироваться от 0 до 50 миллионов пользователей в течение 3 лет. |
| **Почему критично** | Бизнес-цели требуют глобального охвата и роста пользовательской базы (до 50M). Невозможность масштабирования приведёт к отказу системы, потере пользователей и упущенной выручке. |
| **Метрики** | 1. Время отклика API не увеличивается более чем на 20% при росте нагрузки в 3 раза.<br>2. Горизонтальное масштабирование: добавление новых узлов линейно увеличивает пропускную способность.<br>3. Время восстановления после масштабирования < 5 минут.<br>4. Пиковая нагрузка: 10,000 RPS без деградации.<br>5. Обработка данных: 5M тренировок в день. |
| **Архитектурные решения** | 1. Микросервисная архитектура с независимым масштабированием компонентов.<br>2. Горизонтальное масштабирование (Kubernetes HPA).<br>3. Шардирование баз данных (по пользователям, по регионам).<br>4. Асинхронная обработка через Kafka для разгрузки критических путей.<br>5. CQRS для разделения нагрузки на чтение и запись.<br>6. Кеширование (Redis) для снижения нагрузки на БД.<br>7. CDN для статики и медиа.<br>8. Multi-cloud стратегия для географического распределения. |
| **Компромиссы** | 1. Сложность распределённых систем vs простота монолита.<br>2. Eventual Consistency vs строгая консистентность.<br>3. Увеличение затрат на инфраструктуру при росте.<br>4. Сложность отладки в распределённой системе. |
| **Связанные бизнес-цели** | Все (БТ-1, БТ-2, БТ-3, БТ-4) |
| **Связанные ФТ** | Все ФТ, особенно ФТ-1 (запись тренировок) |

---

### 2.2. Надёжность (Reliability / Availability)

| Параметр | Значение |
|:---|:---|
| **Описание** | Способность системы работать без сбоев и отказов, обеспечивая доступность сервисов для пользователей в любое время и в любом месте. Особенно критично во время соревнований и массовых мероприятий. |
| **Почему критично** | Пользователи тренируются в любое время суток; сбой во время тренировки приводит к потере данных и разочарованию. Спортивные события (марафоны) создают пиковые нагрузки, где отказ системы недопустим. |
| **Метрики** | 1. Доступность (SLA): 99.9% (допускается 8.7 часов простоя в год).<br>2. Время восстановления (MTTR): < 30 минут для критических инцидентов.<br>3. Среднее время между сбоями (MTBF): > 30 дней.<br>4. Частота отказов: < 0.01% запросов.<br>5. Сохранность данных: 0% потеря данных (RPO = 0). |
| **Архитектурные решения** | 1. Микросервисная архитектура для изоляции сбоев (сбой в одном сервисе не влияет на другие).<br>2. Репликация баз данных (Master-Slave).<br>3. Автоматическое восстановление (Kubernetes Self-Healing).<br>4. Механизмы повторных попыток (Retry, Exponential Backoff).<br>5. Circuit Breaker для изоляции внешних сбоев.<br>6. Мониторинг и алертинг (Prometheus + Grafana).<br>7. Резервное копирование данных.<br>8. Multi-cloud / Multi-region для географической избыточности. |
| **Компромиссы** | 1. Увеличение сложности vs простота управления.<br>2. Стоимость дополнительных ресурсов (реплики, резервные регионы).<br>3. Eventual Consistency vs строгая согласованность. |
| **Связанные бизнес-цели** | Все (БТ-1, БТ-2, БТ-3, БТ-4) |
| **Связанные ФТ** | ФТ-1 (запись тренировок), ФТ-10 (админ-панель) |

---

### 2.3. Безопасность (Security)

| Параметр | Значение |
|:---|:---|
| **Описание** | Способность системы защищать данные пользователей от несанкционированного доступа, утечек и атак. Особенно критично для данных о здоровье (PHI) и персональных данных (PII), которые подлежат регуляторным требованиям (GDPR, CCPA). |
| **Почему критично** | Утечка данных о здоровье — это не только репутационные потери, но и юридические последствия (штрафы до 4% оборота по GDPR). Пользователи доверяют системе свои самые личные данные, и нарушение этого доверия разрушает бизнес. |
| **Метрики** | 1. Успешное прохождение пентестов: 0 критических уязвимостей.<br>2. Время обнаружения угроз: < 1 минуты.<br>3. Время реагирования на инцидент: < 1 часа.<br>4. 100% шифрование данных в покое и транзите.<br>5. Соответствие GDPR, CCPA, 152-ФЗ: 100% требований реализованы. |
| **Архитектурные решения** | 1. Шифрование данных в покое (AES-256) и транзите (TLS 1.3).<br>2. Аутентификация: OAuth 2.0 + OpenID Connect (Keycloak / Auth0).<br>3. Авторизация: RBAC (роли: пользователь, модератор, партнёр, администратор).<br>4. Шифрование на уровне приложения для PII/PHI.<br>5. Rate Limiting для защиты API.<br>6. WAF (Web Application Firewall) для защиты от DDoS и атак.<br>7. Регулярные пентесты и аудиты безопасности.<br>8. Логирование всех критических действий (Event Store).<br>9. Механизмы "права на забвение" и экспорта данных. |
| **Компромиссы** | 1. Скорость разработки vs безопасность (безопасность требует времени).<br>2. Простота использования vs безопасность (MFA усложняет вход).<br>3. Производительность vs безопасность (шифрование требует ресурсов). |
| **Связанные бизнес-цели** | Все (БТ-1, БТ-2, БТ-3, БТ-4) |
| **Связанные ФТ** | ФТ-12 (публичное API), ФТ-10 (админ-панель), ФТ-2 (подключение устройств) |

## 5. Компромиссы между основными атрибутами качества

| Компромисс | Решение | Обоснование |
|:---|:---|:---|
| **Безопасность vs Производительность** | Шифрование только критических данных; оптимизация алгоритмов шифрования; использование аппаратного ускорения | Безопасность важнее, чем небольшая задержка; пользователи готовы ждать доли секунды ради защиты данных |
| **Надёжность vs Стоимость** | Multi-region только для критических сервисов (Tracking, Users, Auth); для остальных — один регион с репликами | Экономия затрат без потери критической надёжности; некритичные сервисы могут иметь 99.5% SLA |
| **Масштабируемость vs Сложность** | Микросервисная архитектура; внедрение CQRS/Event Sourcing только для высоконагруженных сервисов | Сложность оправдана бизнес-требованиями роста; для простых сервисов допустим монолитный подход |
| **Масштабируемость vs Консистентность** | Eventual Consistency для социальных функций; строгая консистентность для платежей и подписок | Пользователи готовы подождать обновления ленты, но не готовы ждать ошибок при оплате |
