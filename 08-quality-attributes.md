# 8. Атрибуты качества

## 8.1. Масштабируемость

### Описание

Масштабируемость — способность системы выдерживать рост количества пользователей, тренировок, социальных взаимодействий, подключённых устройств и соревнований без существенного ухудшения пользовательского опыта.

Для данного приложения масштабируемость является одним из ключевых атрибутов, так как продукт рассчитан на глобальную аудиторию и должен поддерживать как обычную ежедневную нагрузку, так и кратковременные пики во время соревнований.

### Обоснование

Масштабируемость критична, потому что:
- пользователи находятся в разных регионах мира;
- тренировки и телеметрия создают большой поток данных;
- социальная лента и уведомления создают fan-out нагрузку;
- соревнования могут формировать резкие пики нагрузки;
- разные подсистемы имеют разные профили нагрузки.

Без масштабируемости система будет деградировать при росте аудитории, что напрямую повлияет на вовлечённость, удержание и доверие пользователей.

### Где наиболее критична

- Training Service
- Activity Feed Service
- Notification Service
- Competition Service
- Analytics pipeline
- Event Bus

### Архитектурные решения

- stateless сервисы;
- горизонтальное масштабирование;
- autoscaling;
- event-driven обработка;
- кэширование read-heavy сценариев;
- partitioning / sharding данных;
- разделение transactional и analytical контуров.

### Показатели / метрики

| Метрика | Целевое значение |
|---|---|
| P95 latency для основных API | < 200 мс |
| P99 latency для критичных API | < 500 мс |
| Throughput записи тренировок | масштабируется горизонтально |
| Время обработки события `training.completed` | < 5 сек для стандартных сценариев |
| Время обновления leaderboard | < 200 мс для чтения |
| Утилизация CPU сервисов | < 70% в steady state |
| Время масштабирования сервиса | < 5 минут |
| Доля ошибок при пиковых нагрузках | < 1% |

## 8.2. Надёжность

### Описание

Надёжность — способность системы корректно работать в условиях сбоев, частичной недоступности сервисов, сетевых ошибок и повторной доставки сообщений.

Для фитнес-приложения надёжность особенно важна в сценариях записи тренировок, обработки событий и сохранения пользовательского прогресса.

### Обоснование

Потеря тренировочных данных напрямую снижает доверие пользователя. Если пользователь завершил тренировку, но система её не сохранила, это воспринимается как критическая ошибка, даже если остальные функции продолжают работать.

Надёжность также важна для:
- корректного начисления достижений;
- обновления прогресса;
- отправки уведомлений;
- обработки данных с устройств;
- восстановления после сбоев.

### Где наиболее критична

- Training Service
- Device Integration Service
- Event Bus
- Analytics Service
- Notification Service
- Competition Service

### Архитектурные решения

- idempotent API;
- retry-механизмы;
- durable queues;
- outbox pattern;
- DLQ для ошибочных событий;
- репликация данных;
- резервное копирование;
- graceful degradation;
- circuit breaker для внешних интеграций.

### Показатели / метрики

| Метрика | Целевое значение |
|---|---|
| Успешность записи тренировок | > 99.9% |
| Потеря событий | 0 для критичных бизнес-событий |
| Доля успешно обработанных событий | > 99.9% |
| RPO для тренировочных данных | ≤ 5 минут |
| RTO для основных сервисов | ≤ 30 минут |
| Availability core API | ≥ 99.9% |
| Доля повторно обработанных событий без дублей | 100% для идемпотентных операций |
| Время восстановления после сбоя | в пределах RTO |

## 8.3. Безопасность

### Описание

Безопасность — способность системы защищать пользовательские данные, управлять доступом, предотвращать несанкционированные действия и обеспечивать выполнение регуляторных требований.

Система обрабатывает чувствительные данные:
- персональные данные;
- данные о здоровье и физической активности;
- маршруты и геолокацию;
- данные устройств;
- социальные связи.

### Обоснование

Безопасность является одним из ключевых атрибутов, потому что нарушение конфиденциальности может привести к:
- потере доверия пользователей;
- юридическим последствиям;
- штрафам;
- репутационному ущербу;
- ограничению работы продукта в отдельных регионах.

Особое внимание требуется уделить приватности маршрутов, текущей геолокации и медицински чувствительным данным.

### Где наиболее критична

- Identity & Access Service
- User Profile Service
- Training Service
- Device Integration Service
- Social Graph Service
- Commerce Integration Service
- Analytics / Data Platform

### Архитектурные решения

- OAuth2 / OIDC;
- RBAC / ABAC;
- шифрование данных при передаче и хранении;
- централизованное управление секретами;
- аудит доступа к чувствительным данным;
- privacy by design;
- consent management;
- data classification;
- минимизация данных;
- маскирование геолокации;
- WAF и rate limiting.

### Показатели / метрики

| Метрика | Целевое значение |
|---|---|
| Доля зашифрованных чувствительных данных | 100% |
| Доля API под аутентификацией | 100% |
| Доля критичных действий с audit log | 100% |
| Время отзыва токена / сессии | < 5 минут |
| Количество критичных уязвимостей в production | 0 |
| SLA обработки запроса на удаление данных | в рамках регуляторных требований |
| Доля сервисов с managed secrets | 100% |
| Доля публично доступных чувствительных данных по умолчанию | 0% |

## 8.4. Обоснование выбора атрибутов качества

Для данной системы основными выбраны три атрибута качества:

1. масштабируемость;
2. надёжность;
3. безопасность.

Именно эти атрибуты напрямую следуют из бизнес- и продуктового контекста.

### Почему выбрана масштабируемость

Продукт рассчитан на глобальную аудиторию и потенциально массовые соревнования. Нагрузка будет неоднородной: обычные тренировки создают постоянный поток данных, а соревнования — резкие пики. Поэтому архитектура должна масштабироваться горизонтально и позволять отдельно масштабировать наиболее нагруженные компоненты.

### Почему выбрана надёжность

Основная пользовательская ценность связана с записью тренировок и сохранением прогресса. Потеря данных даже у одного пользователя может привести к потере доверия. Поэтому критичные операции должны быть устойчивыми к сбоям, повторной доставке сообщений и частичной недоступности зависимостей.

### Почему выбрана безопасность

Приложение обрабатывает чувствительные данные: здоровье, маршруты, геолокацию, социальные связи и данные устройств. Для глобального продукта это создаёт высокие юридические и репутационные риски. Поэтому безопасность и приватность должны быть встроены в архитектуру с самого начала.

### Почему остальные атрибуты не выбраны как основные

Такие атрибуты, как maintainability, usability, performance и observability, также важны, но в данном проекте они являются поддерживающими.

- **Performance** частично покрывается масштабируемостью и конкретными latency-метриками.
- **Observability** необходима для обеспечения надёжности микросервисной архитектуры.
- **Maintainability** обеспечивается доменной декомпозицией, контрактами API и CI/CD.
- **Usability** важна для продукта, но в архитектурной части является следствием правильной работы core-сценариев.

Таким образом, выбранные атрибуты качества отражают основные архитектурные риски и бизнес-приоритеты системы.
