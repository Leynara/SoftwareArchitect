# №13: Базовая архитектура с учётом ограничений бизнес-требований, НФТ, выбранной архитектуры, адресация атрибутов качества

---

**Назначение раздела:** Базовая архитектура объединяет все ранее принятые архитектурные решения (№10) и показывает, как система удовлетворяет бизнес-целям, функциональным и нефункциональным требованиям, а также ограничениям. Раздел демонстрирует, как атрибуты качества (масштабируемость, надёжность, безопасность) обеспечиваются на уровне архитектуры, и как учитываются ограничения (multi-cloud, GDPR, интеграция с legacy, офлайн-режим).

| Ограничение (из №1) | Реализация в архитектуре | Архитектурное решение |
|:---|:---|:---|
| **C-01 (GDPR/CCPA)** | Географическое шардирование данных; шифрование PII/PHI на уровне приложения; механизмы удаления/экспорта данных | ADR-005 (Шардирование), ADR-011 (Безопасность) |
| **C-02 (Multi-cloud)** | Kubernetes + открытые стандарты; Terraform + Crossplane; избегание проприетарных сервисов | ADR-008 (Cloud-agnostic), ADR-010 (Kubernetes) |
| **C-03 (Legacy-интеграция)** | Anti-Corruption Layer; API Gateway с трансформацией; поэтапная интеграция | ADR-003 (API Gateway) |
| **C-04 (Бюджет: $5M на первый год)** | Оптимизация затрат: open-source инструменты, spot-инстансы, автоматическое архивирование данных | ADR-009 (Open-source observability) |
| **C-05 (Сроки: MVP за 9 месяцев)** | Приоритизация функциональности (ФТ-1..ФТ-7), use of BaaS (Auth0 для аутентификации) | ADR-007 (Rule-based AI), ADR-011 (Auth0) |
| **C-06 (Языковой барьер)** | Единый стандарт кодирования; документация на английском; API-first подход | ADR-003, ADR-010 |
| **C-07 (Офлайн-режим)** | Локальная БД (SQLite/Realm) на устройстве; Event Sourcing для разрешения конфликтов; CRDT-паттерны | ADR-001 (Event Sourcing) |

## 4. Адресация атрибутов качества

### 4.1. Масштабируемость (Scalability)

| Требование | Реализация | Архитектурное решение |
|:---|:---|:---|
| 10,000 RPS пиковая нагрузка | Микросервисы с независимым масштабированием; Kafka для асинхронности; кеширование | ADR-001, ADR-002, ADR-004 |
| 50M пользователей через 3 года | Шардирование БД (по пользователю и региону), горизонтальное масштабирование | ADR-005 |
| 5M тренировок в день | TimescaleDB с партиционированием; Kafka для потоковой обработки | ADR-006 |
| Линейное масштабирование | Kubernetes HPA; использование stateless сервисов | ADR-010 |

### 4.2. Надёжность (Reliability / Availability)

| Требование | Реализация | Архитектурное решение |
|:---|:---|:---|
| 99.9% SLA | Репликация БД (Master-Slave), Multi-region развертывание | ADR-001, ADR-010 |
| MTTR < 30 минут | Автоматическое восстановление (Kubernetes), мониторинг, алертинг | ADR-009, ADR-010 |
| 0% потеря данных (RPO=0) | Репликация, резервное копирование, Event Sourcing | ADR-001, ADR-004 |
| Отказ одного сервиса не влияет на другие | Микросервисная архитектура, Circuit Breaker, Bulkhead | ADR-001, ADR-002 |

### 4.3. Безопасность (Security)

| Требование | Реализация | Архитектурное решение |
|:---|:---|:---|
| Шифрование данных в покое и транзите | AES-256, TLS 1.3, шифрование на уровне приложения | ADR-011 |
| 0 критических уязвимостей | Регулярные пентесты, аудит, OAuth 2.0 + JWT | ADR-011 |
| Соответствие GDPR/CCPA | Шардирование по регионам, удаление/экспорт данных, логирование действий | ADR-005, ADR-011 |
| Защита API от атак | Rate Limiting, WAF, IP Whitelisting, JWT валидация | ADR-003, ADR-011 |

## 7. Компромиссы в базовой архитектуре

| Компромисс | Решение | Причина |
|:---|:---|:---|
| **Сложность vs Гибкость** | Микросервисы | Требование независимой разработки команд; эволюционируемость системы |
| **Консистентность vs Производительность** | Eventual Consistency, асинхронность | Глобальное распределение; пиковые нагрузки (марафоны) |
| **Multi-cloud vs Простота** | Cloud-agnostic подход | Бизнес-требование отсутствия vendor lock-in |
| **Безопасность vs Простота использования** | MFA опционально | Баланс между безопасностью и юзабилити |
| **Стоимость vs Производительность** | Open-source инструменты | Снижение затрат при сохранении качества |
| **Офлайн-режим vs Сложность синхронизации** | Event Sourcing + CRDT | Критичное требование для пользователей |
