# №13: Базовая архитектура с учётом ограничений бизнес-требований, НФТ, выбранной архитектуры, адресация атрибутов качества

---

**Назначение раздела:** Базовая архитектура объединяет все ранее принятые архитектурные решения (№10) и показывает, как система удовлетворяет бизнес-целям, функциональным и нефункциональным требованиям, а также ограничениям. Раздел демонстрирует, как атрибуты качества (масштабируемость, надёжность, безопасность) обеспечиваются на уровне архитектуры, и как учитываются ограничения (multi-cloud, GDPR, интеграция с legacy, офлайн-режим).

| **7. Внешние интеграции** | Интеграция с внешними системами | Anti-Corruption Layer, API-адаптеры | Адаптация legacy-систем, платежи, импорт данных, карты, погода |

### 1.3. Связь компонентов с ограничениями

| Ограничение | Реализация в архитектуре | Компоненты |
|:---|:---|:---|
| **C-01 (GDPR/CCPA)** | Географическое шардирование данных; шифрование PII/PHI на уровне приложения; механизмы удаления/экспорта данных | PostgreSQL (шардирование по регионам), Сервис Users (управление данными), Keycloak (аутентификация) |
| **C-02 (Multi-cloud)** | Kubernetes + открытые стандарты; Terraform + Crossplane; избегание проприетарных сервисов | Kubernetes, PostgreSQL, Kafka, Redis, Elasticsearch, TimescaleDB, Neo4j |
| **C-03 (Legacy-интеграция)** | Anti-Corruption Layer; API Gateway с трансформацией; поэтапная интеграция | Anti-Corruption Layer, API Gateway, Сервис Commerce |
| **C-04 (Бюджет: $5M)** | Оптимизация затрат: open-source инструменты, spot-инстансы, автоматическое архивирование | Prometheus, Grafana, ELK, Jaeger, Kubernetes, TimescaleDB (сжатие) |
| **C-05 (Сроки: MVP за 9 месяцев)** | Приоритизация функциональности (ФТ-1..ФТ-7), use of BaaS (Auth0 для аутентификации) | Auth0/Keycloak, Rule-based AI (MVP), упрощённая архитектура без CQRS на старте |
| **C-06 (Языковой барьер)** | Единый стандарт кодирования; документация на английском; API-first подход | OpenAPI/Swagger, GitLab CI, ADR-документация на английском |
| **C-07 (Офлайн-режим)** | Локальная БД (SQLite/Realm) на устройстве; Event Sourcing для разрешения конфликтов; CRDT-паттерны | Мобильное приложение (SQLite), Сервис Tracking (Event Sourcing) |

## 3. Адресация атрибутов качества

### 3.1. Масштабируемость

| Требование | Реализация | Компоненты |
|:---|:---|:---|
| 10,000 RPS пиковая нагрузка | Микросервисы с независимым масштабированием; Kafka для асинхронности; кеширование | API Gateway, Сервис Tracking, Kafka, Redis |
| 50M пользователей через 3 года | Шардирование БД (по пользователю и региону), горизонтальное масштабирование | PostgreSQL (шардирование), TimescaleDB (партиционирование) |
| 5M тренировок в день | TimescaleDB с партиционированием; Kafka для потоковой обработки | TimescaleDB, Kafka, Сервис Tracking |
| Линейное масштабирование | Kubernetes HPA; использование stateless сервисов | Kubernetes, все микросервисы |

### 3.2. Надёжность

| Требование | Реализация | Компоненты |
|:---|:---|:---|
| 99.9% SLA | Репликация БД (Master-Slave), Multi-region развертывание | PostgreSQL (репликация), Kubernetes (Multi-region) |
| MTTR < 30 минут | Автоматическое восстановление (Kubernetes), мониторинг, алертинг | Kubernetes, Prometheus, Grafana |
| 0% потеря данных (RPO=0) | Репликация, резервное копирование, Event Sourcing | PostgreSQL (репликация), Event Store, Kafka |
| Отказ одного сервиса не влияет на другие | Микросервисная архитектура, Circuit Breaker, Bulkhead | Все микросервисы, Istio (Service Mesh) |

### 3.3. Безопасность

| Требование | Реализация | Компоненты |
|:---|:---|:---|
| Шифрование данных в покое и транзите | AES-256, TLS 1.3, шифрование на уровне приложения | PostgreSQL (шифрование), API Gateway (TLS), Keycloak |
| 0 критических уязвимостей | Регулярные пентесты, аудит, OAuth 2.0 + JWT | Keycloak, Сервис Users (RBAC) |
| Соответствие GDPR/CCPA | Шардирование по регионам, удаление/экспорт данных, логирование действий | PostgreSQL (шардирование), Сервис Users (удаление данных), Event Store (аудит) |
| Защита API от атак | Rate Limiting, WAF, IP Whitelisting, JWT валидация | API Gateway, WAF (Cloudflare/AWS Shield) |

## 4. Компромиссы в базовой архитектуре

| Компромисс | Решение | Причина |
|:---|:---|:---|
| **Сложность vs Гибкость** | Микросервисы | Требование независимой разработки команд; эволюционируемость системы |
| **Консистентность vs Производительность** | Eventual Consistency, асинхронность | Глобальное распределение; пиковые нагрузки (марафоны) |
| **Multi-cloud vs Простота** | Cloud-agnostic подход | Бизнес-требование отсутствия vendor lock-in |
| **Безопасность vs Простота использования** | MFA опционально | Баланс между безопасностью и юзабилити |
| **Стоимость vs Производительность** | Open-source инструменты | Снижение затрат при сохранении качества |
| **Офлайн-режим vs Сложность синхронизации** | Event Sourcing + CRDT | Критичное требование для пользователей |
